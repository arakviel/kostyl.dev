# Синхронізація: lock, Monitor та Critical Section

## Від Проблеми до Рішення

Попередня тема (05/05а) детально описала природу проблем спільного стану: чому `i++` не атомарний, як CPU кеш "ховає" зміни, і чому check-then-act без синхронізації — це Recipe for Disaster. Тепер — найважливіше питання курсу: **як правильно захистити спільний стан?**

Відповідь на це питання формується навколо єдиної центральної концепції — **взаємного виключення** (Mutual Exclusion). Якщо тільки один потік може виконувати певний код одночасно, жодна з описаних проблем не виникне. Саме це і гарантує `lock`.

## Critical Section: Концепція

### Що Таке Критична Секція

**Критична секція** (Critical Section) — це ділянка коду, де відбувається доступ до спільних ресурсів і де **неправильне чергування** операцій між потоками може призвести до некоректного стану.

Ключові характеристики критичної секції:
- Вона містить доступ до **спільного (shared) стану** — поля об'єкта, глобальної змінної, зовнішнього файлу, мережевого з'єднання
- У ній відбувається **читання-модифікація-запис** або інші non-atomic операції
- Вхід у неї повинен бути **виключним**: якщо один потік всередині — інші мусять чекати

![Critical Section — взаємне виключення](/images/system-programming/06/critical-section.png)

Аналогія: уявіть одномісну вбиральню з замком на дверях. Якщо всередині хтось є — на дверях висить табличка "Зайнято". Ніхто інший не може увійти. Як тільки перший вийде і зніме замок — наступний з черги може зайти. Ніякого "паралельного" перебування всередині: **mutual exclusion** (взаємне виключення).

### Властивості Коректної Синхронізації

Академічна теорія синхронізації формулює три ключові вимоги:

1. **Mutual Exclusion** (Взаємне виключення): у будь-який момент часу не більше `K` потоків перебуває у критичній секції (для звичайного lock: `K=1`, для semaphore: `K=N`)
2. **Progress** (Прогрес): якщо критична секція вільна і є потоки, що хочуть увійти — хтось з них таки увійде (немає безкінечного зависання через сам механізм блокування)
3. **Bounded Waiting** (Обмежене очікування): кожен потік, що хоче увійти, зробить це за скінченний час (немає Starvation)

.NET `lock`/`Monitor` гарантує перші два. Третє (Starvation freedom) — залежить від планувальника ОС і не гарантується специфікацією .NET (хоча на практиці CLR використовує fair queue для Monitor).

## `lock` Statement: Синтаксис та Правила

### Основний Синтаксис

```csharp showLineNumbers [LockBasics.cs]
using System.Threading;

public class ThreadSafeBankAccount
{
    // ✅ ЗОЛОТИЙ СТАНДАРТ: приватний, readonly, окремий об'єкт виключно для lock
    private readonly object _syncRoot = new object();

    private decimal _balance;

    public ThreadSafeBankAccount(decimal initialBalance)
    {
        _balance = initialBalance;
    }

    public bool TryWithdraw(decimal amount)
    {
        // lock(_syncRoot): отримати "ексклюзивне право" на _syncRoot
        // якщо _syncRoot вже зайнятий іншим потоком — поточний потік БЛОКУЄТЬСЯ
        // і чекає поки _syncRoot не стане доступним
        lock (_syncRoot)
        {
            // ── КРИТИЧНА СЕКЦІЯ ──────────────────────────────────
            // Тут гарантовано: лише один потік одночасно
            
            if (_balance < amount)
                return false;

            // Між if і -= нема вікна вразливості — все в одному lock
            _balance -= amount;
            return true;
            // ── КІНЕЦЬ КРИТИЧНОЇ СЕКЦІЇ ──────────────────────────
        }
        // Ось тут lock автоматично звільнено (навіть при return/exception)
    }

    public void Deposit(decimal amount)
    {
        lock (_syncRoot)  // той самий _syncRoot → взаємно виключає з TryWithdraw
        {
            _balance += amount;
        }
    }

    // Не потрібно lock: decimal читання thread-safe на x64 (per spec: ≤ 8 bytes)
    // АЛЕ для узгодженості — завжди читайте через lock якщо є write-ers
    public decimal Balance
    {
        get { lock (_syncRoot) { return _balance; } }
    }
}
```

### Критичні Правила Вибору Lock Object

Вибір об'єкта для lock — **не технічне, а архітектурне рішення**. Неправильний вибір призведе до bugs, що дуже важко знайти.

| Lock Object | Оцінка | Чому |
| :--- | :--- | :--- |
| `private readonly object _lock = new()` | ✅ **Правильно** | Повний контроль, недоступний зовні, призначений виключно для lock |
| `private readonly object _lock` | ✅ **Правильно** | Аналогічно (new призначається в конструкторі) |
| `lock(this)` | ❌ **Ніколи** | `this` доступний зовні — зовнішній код може `lock(obj)` той самий об'єкт |
| `lock(typeof(MyClass))` | ❌ **Ніколи** | `Type` об'єкт — глобальний singleton. Lock на тип → блокує ВСЮ AppDomain. Race condition між класами |
| `lock("constant string")` | ❌ **Ніколи** | String literals інтеруються → два незв'язані класи з однаковим рядком блокують одне одного |
| `lock(new object())` | ❌ **Ніколи** | Новий об'єкт кожен раз → ніякого взаємного виключення, просто дорога no-op |
| `private static readonly object _lock` | ⚠️ **Обережно** | Static lock = один для всіх екземплярів. Виправдано для static state, але занадто широкий scope для instance state |
| `lock(SomeMutableField)` | ❌ **Ніколи** | Якщо поле перепризначується — два потоки можуть lock-нути РІЗНІ об'єкти |

```csharp showLineNumbers [LockObjectPitfalls.cs]
public class Counter
{
    // ✅ правильно
    private readonly object _lock = new object();

    // ❌ Не можна lock(this) — this доступний назовні
    public void WrongLockThis()
    {
        lock (this)  // ← зовнішній код може: lock(myCounter) { ... }
        { _count++; }
    }

    // ❌ Не можна lock(typeof)
    public void WrongLockType()
    {
        lock (typeof(Counter))  // ← блокує ДО тип для OperatingSystem
        { _count++; }
    }

    // ❌ Не можна lock на нестатичну string
    private string _name = "shared_resource";
    public void WrongLockString()
    {
        lock (_name)  // ← якщо _name змінюється — катастрофа
        { _count++; }
    }

    // ✅ правильно — ізольований, незмінний, private
    public void Increment()
    {
        lock (_lock) { _count++; }
    }

    private int _count;
}
```

### Lock Granularity: Грубий vs Тонкий

**Lock granularity** — наскільки велику частину стану захищає один lock:

- **Coarse-grained lock** (грубий): один lock захищає весь об'єкт або навіть кілька об'єктів. Простий у реалізації, менший ризик deadlock. Але — знижує паралелізм: навіть незв'язані операції блокують одна одну.

- **Fine-grained lock** (тонкий): окремий lock для кожного незалежного поля або ресурсу. Максимальний паралелізм, але складніший код і вищий ризик deadlock.

```csharp showLineNumbers [LockGranularity.cs]
// ❌ Coarse-grained: один lock на весь об'єкт
public class UserCache_Coarse
{
    private readonly object _lock = new();
    private Dictionary<int, User> _byId = new();
    private Dictionary<string, User> _byEmail = new();

    public void Add(User u)
    {
        lock (_lock)  // блокує і byId і byEmail операції одночасно
        {
            _byId[u.Id] = u;
            _byEmail[u.Email] = u;
        }
    }

    public User? GetById(int id)
    {
        lock (_lock) { return _byId.GetValueOrDefault(id); }
    }

    public User? GetByEmail(string email)
    {
        lock (_lock) { return _byEmail.GetValueOrDefault(email); }
    }
    // Читання GetById і GetByEmail блокують одне одного — нема паралелізму!
}

// ✅ Fine-grained: окремі lock для незалежних колекцій
public class UserCache_Fine
{
    private readonly object _idLock = new();
    private readonly object _emailLock = new();
    private Dictionary<int, User> _byId = new();
    private Dictionary<string, User> _byEmail = new();

    public void Add(User u)
    {
        lock (_idLock)    { _byId[u.Id] = u; }
        lock (_emailLock) { _byEmail[u.Email] = u; }
        // ОБЕРЕЖНО: між двома lock-ами є вікно де byId має User але byEmail ще ні
        // Якщо потрібна узгодженість між двома — потрібно захопити обидва одночасно
    }

    public User? GetById(int id)
    {
        lock (_idLock) { return _byId.GetValueOrDefault(id); }
        // GetByEmail може виконуватись паралельно! ✓
    }
}
```

## Під Капотом: lock → Monitor

### Що Генерує Компілятор

Ключове слово `lock` — це **синтаксичний цукор**. Компілятор C# розгортає його у виклики `Monitor.Enter` та `Monitor.Exit`, оборнені у `try/finally`:

![lock → Monitor під капотом](/images/system-programming/06/lock-to-monitor.png)

```csharp showLineNumbers [LockDecompiled.cs]
// ВАШ КОД:
lock (_syncRoot)
{
    _balance -= amount;
}

// ЩО ГЕНЕРУЄ КОМПІЛЯТОР (спрощено):
bool lockTaken = false;
try
{
    Monitor.Enter(_syncRoot, ref lockTaken);
    // ↑ lockTaken = true якщо lock успішно захоплено
    _balance -= amount;
}
finally
{
    // finally виконується ЗАВЖДИ — навіть при return, Exception, ThreadAbortException
    if (lockTaken)
        Monitor.Exit(_syncRoot);
    // ↑ lock ЗАВЖДИ звільняється
}
```

**Чому `lockTaken` bool?** Уявіть ситуацію: `Monitor.Enter` почав виконуватись, але між ним та захопленням lock стався `OutOfMemoryException`. Без `lockTaken` — ми не знали б чи lock вже захоплено. З `ref lockTaken` — `Monitor.Enter` атомарно встановлює його before гарантованого захоплення, і `finally` може безпечно перевірити чи потрібно звільняти.

**Чому `finally`?** Це гарантія звільнення lock навіть при будь-якому виключенні. Без `finally` — якщо в критичній секції виникне `Exception`, lock залишиться назавжди захопленим → всі інші потоки будуть заблоковані → deadlock.

### Monitor — Повний Клас

`System.Threading.Monitor` надає більше контролю ніж `lock`:

```csharp showLineNumbers [MonitorApi.cs]
using System.Threading;

object _syncRoot = new object();

// ─── Базовий Enter/Exit ───────────────────────────────────────
Monitor.Enter(_syncRoot);           // Блокуючий Enter
try { /* критична секція */ }
finally { Monitor.Exit(_syncRoot); }  // Завжди звільняємо

// ─── TryEnter: спроба без нескінченного очікування ───────────
bool acquired = Monitor.TryEnter(_syncRoot);             // без timeout
bool acquired2 = Monitor.TryEnter(_syncRoot, 500);       // ms
bool acquired3 = Monitor.TryEnter(_syncRoot, TimeSpan.FromSeconds(1));

if (acquired)
{
    try { /* критична секція */ }
    finally { Monitor.Exit(_syncRoot); }
}
else
{
    // lock не отриманий за 500ms → fallback логіка
    Console.WriteLine("Не вдалось отримати lock — спробуємо пізніше");
}

// ─── IsEntered: чи поточний потік тримає цей lock ─────────────
bool iHaveIt = Monitor.IsEntered(_syncRoot);
// Корисно для assertion/guard у методах
```

## Monitor.Wait та Monitor.Pulse: Сигналізація

### Навіщо Wait/Pulse

`lock` — це взаємне виключення. Але часто потоки потребують не просто виключення, а **координації**: один потік чекає поки в іншого з'явяться дані або виконається умова. Для цього Monitor надає **condition variable API**:

- `Monitor.Wait(lock)` — звільняє lock і засинає (атомарно!), чекаючи сигналу. Поки потік спить — інші можуть захопити lock.
- `Monitor.Pulse(lock)` — будить **один** потік з wait-черги для цього lock object.
- `Monitor.PulseAll(lock)` — будить **всі** потоки з wait-черги.

![Monitor.Wait / Pulse — сигналізація](/images/system-programming/06/wait-pulse.png)

### Інваріант: `while` замість `if` перед Wait

Найважливіше правило використання Wait/Pulse:

```csharp showLineNumbers [WaitPulsePattern.cs]
// ❌ НЕПРАВИЛЬНО: if перед Wait
lock (_queue)
{
    if (_queue.Count == 0)       // перевіряємо умову
        Monitor.Wait(_lock);     // засинаємо

    var item = _queue.Dequeue(); // ← Spurious wakeup! Queue може ще бути порожньою
}

// ✅ ПРАВИЛЬНО: while перед Wait
lock (_queue)
{
    while (_queue.Count == 0)   // ЗАВЖДИ while, ніколи if!
        Monitor.Wait(_lock);    // перевіримо знову після пробудження

    var item = _queue.Dequeue();
}
```

**Spurious Wakeup** (хибне пробудження) — стан де потік прокидається з Wait() без явного Pulse(). Це документована поведінка POSIX pthreads (на якому реалізований CLR на Linux) і теоретично можлива на будь-якій платформі. `while` гарантує що умова справді виконалась після пробудження.

### Blocking Queue через Wait/Pulse

Класичний приклад координації — блокуюча черга (BlockingQueue): Producer додає елементи, Consumer забирає, Consumer блокується якщо черга порожня:

```csharp showLineNumbers [BlockingQueue.cs]
using System.Collections.Generic;
using System.Threading;

public class BlockingQueue<T>
{
    private readonly Queue<T> _queue = new();
    private readonly int _maxCapacity;
    private readonly object _lock = new();

    public BlockingQueue(int maxCapacity = int.MaxValue)
    {
        _maxCapacity = maxCapacity;
    }

    /// <summary>
    /// Додати елемент. Якщо черга повна — БЛОКУЄТЬСЯ до появи місця.
    /// </summary>
    public void Enqueue(T item)
    {
        lock (_lock)
        {
            // Чекаємо поки є місце (може бути spurious wakeup → while)
            while (_queue.Count >= _maxCapacity)
                Monitor.Wait(_lock);

            _queue.Enqueue(item);

            // Будимо Consumer якщо він чекав на порожній черзі
            Monitor.PulseAll(_lock);
            // PulseAll: будимо і Consumer-ів і Producer-ів
            // (можуть чекати обидва типи коли одночасно і повна, і є споживачі)
        }
    }

    /// <summary>
    /// Вилучити елемент. Якщо черга порожня — БЛОКУЄТЬСЯ до появи елементу.
    /// </summary>
    public T Dequeue()
    {
        lock (_lock)
        {
            // Чекаємо поки є хоча б один елемент
            while (_queue.Count == 0)
                Monitor.Wait(_lock);  // атомарно: звільняє _lock + засинає

            T item = _queue.Dequeue();

            // Будимо Producer якщо він чекав через повну чергу
            Monitor.PulseAll(_lock);

            return item;
        }
    }

    /// <summary>
    /// Спроба вилучити без блокування → false якщо порожньо
    /// </summary>
    public bool TryDequeue(out T item)
    {
        lock (_lock)
        {
            if (_queue.Count == 0)
            {
                item = default!;
                return false;
            }
            item = _queue.Dequeue();
            Monitor.Pulse(_lock);  // Один Producer достатньо
            return true;
        }
    }

    public int Count { get { lock (_lock) { return _queue.Count; } } }
}
```

```csharp showLineNumbers [BlockingQueueDemo.cs]
var queue = new BlockingQueue<int>(capacity: 10);
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));

// Producer: генерує числа 1..100
var producer = Task.Run(() =>
{
    for (int i = 1; i <= 100 && !cts.Token.IsCancellationRequested; i++)
    {
        queue.Enqueue(i);
        Console.WriteLine($"[P] Enqueued: {i,3} | QueueSize: {queue.Count}");
        Thread.Sleep(20);  // імітуємо затримку при генерації
    }
});

// Consumer: обробляє елементи (повільніший за Producer)
var consumer = Task.Run(() =>
{
    while (!cts.Token.IsCancellationRequested)
    {
        int item = queue.Dequeue();  // блокується якщо черга порожня
        Console.WriteLine($"       [C] Деqueued: {item,3}");
        Thread.Sleep(50);  // Consumer вдвічі повільніший
    }
});

await Task.WhenAll(producer, consumer);
```

## `System.Threading.Lock` (C# 13 / .NET 9)

### Новий Тип: Оптимізований Lock

.NET 9 (C# 13) представив `System.Threading.Lock` — новий dedicated тип для синхронізації, що замінює старий підхід з `object`:

```csharp showLineNumbers [NewLockType.cs]
using System.Threading;

// Старий підхід: object як lock
private readonly object _oldLock = new object();

// ✅ Новий підхід у .NET 9: System.Threading.Lock
private readonly Lock _lock = new Lock();

// Базове використання — так само як object:
lock (_lock)  // компілятор знає що це Lock, не object → оптимізований шлях
{
    // критична секція
}

// Новий API: EnterScope() — ref struct з IDisposable
// Переваги: видно у IDE як resource, гарантія Release через using
using (var scope = _lock.EnterScope())
{
    // критична секція
    // scope.Dispose() при виході = _lock.Exit()
}

// Ще коротше з var:
{
    using var _ = _lock.EnterScope();
    // критична секція
}

// TryEnter — як Monitor.TryEnter
if (_lock.TryEnter(out var scope))
{
    using (scope)
    {
        // захоплено, виконуємо
    }
}
```

### Порівняння: `object` vs `Lock`

| Характеристика | `object` lock | `System.Threading.Lock` |
| :--- | :--- | :--- |
| .NET версія | Будь-яка | .NET 9+ |
| Тип | Будь-який `object` | Спеціалізований тип |
| Продуктивність | Базова | Оптимізована (менше allocations) |
| `EnterScope()` | ❌ Ні | ✅ ref struct з using |
| `TryEnter(scope)` | Через `Monitor.TryEnter` | Вбудований |
| Lock recursion | Через `Monitor` | Не підтримує (intentionally) |
| Thread Ownership | Через Monitor | Вбудована перевірка |
| Stack trace | Менш зрозумілий | Краще іменований |

::note
Якщо ваш проект на .NET 9+, `System.Threading.Lock` — рекомендований підхід для нових класів. Для підтримки старих версій продовжуйте використовувати `private readonly object _lock = new()`. Семантика однакова — компілятор розгортає `lock(_lock)` в оптимізований код якщо `_lock` є `Lock` типом.
::

## Deadlock: Взаємне Блокування

### Визначення та Чотири Умови Coffman

**Deadlock** — стан системи де два або більше потоків **заблоковані навічно**, кожен чекаючи ресурс, що утримується іншим. Процес не "зависає" у сенсі CPU (CPU = 0%), але і не прогресує.

Для виникнення deadlock необхідні **чотири умови Coffman** (всі чотири одночасно):

1. **Mutual Exclusion**: ресурс може бути захоплений лише одним потоком одночасно
2. **Hold-and-Wait**: потік тримає один ресурс і чекає на інший
3. **No Preemption**: ресурс не можна відібрати примусово — тільки добровільно звільнити
4. **Circular Wait**: існує цикл взаємних залежностей (A чекає B, B чекає A)

![Deadlock — два потоки, два locks](/images/system-programming/06/deadlock.png)

Якщо прибрати **будь-яку** з чотирьох умов — deadlock стає неможливим. Саме це і є стратегіями запобігання.

### Відтворення Deadlock

```csharp showLineNumbers [DeadlockDemo.cs]
using System;
using System.Threading;

// Два lock-об'єкти, два потоки, зворотній порядок захоплення
object lockA = new object();
object lockB = new object();

// Thread 1: захоплює A, потім намагається захопити B
var thread1 = new Thread(() =>
{
    lock (lockA)
    {
        Console.WriteLine("[T1] Захопив lockA, чекаю lockB...");
        Thread.Sleep(100);  // імітуємо затримку, даємо Thread2 захопити lockB

        lock (lockB)  // ← ЗАБЛОКУЄТЬСЯ НАЗАВЖДИ якщо T2 вже тримає lockB
        {
            Console.WriteLine("[T1] Захопив обидва! (ніколи не виконається)");
        }
    }
}) { Name = "Thread1" };

// Thread 2: захоплює B, потім намагається захопити A (зворотній порядок)
var thread2 = new Thread(() =>
{
    lock (lockB)  // ← захоплює B першим (протилежно Thread1)
    {
        Console.WriteLine("[T2] Захопив lockB, чекаю lockA...");
        Thread.Sleep(100);

        lock (lockA)  // ← ЗАБЛОКУЄТЬСЯ НАЗАВЖДИ (T1 тримає lockA)
        {
            Console.WriteLine("[T2] Захопив обидва! (ніколи не виконається)");
        }
    }
}) { Name = "Thread2" };

thread1.Start();
thread2.Start();

// Обидва потоки заблоковані. Приєднання чекатиме вічно:
thread1.Join(); // ← ніколи не поверне контроль
thread2.Join();
```

### Стратегія 1: Lock Ordering (Найефективніша)

Усуваємо умову **Circular Wait**: завжди захоплюємо locks в **одному глобальному порядку**:

![Lock Ordering — запобігання deadlock](/images/system-programming/06/lock-ordering.png)

```csharp showLineNumbers [LockOrdering.cs]
// ✅ Lock Ordering: завжди lockA перед lockB, незалежно від того, хто запитує
object lockA = new object();
object lockB = new object();

// Thread 1: захоплює в порядку A → B
void Thread1Work()
{
    lock (lockA) { lock (lockB) { /* робота */ } }
}

// Thread 2: ТАК САМО A → B (не B → A!)
void Thread2Work()
{
    lock (lockA) { lock (lockB) { /* інша робота */ } }
    // Thread 2 може заблокуватись чекаючи lockA якщо Thread1 тримає
    // Але Thread1 ЗАВЖДИ звільнить lockA, тому deadlock неможливий
}

// ---
// Для динамічних об'єктів: використовуємо hashCode або id для визначення порядку
public static void TransferFunds(Account from, Account to, decimal amount)
{
    // Визначаємо порядок за hashCode — гарантований глобальний порядок
    object first  = RuntimeHelpers.GetHashCode(from) < RuntimeHelpers.GetHashCode(to) ? from : to;
    object second = ReferenceEquals(first, from) ? to : from;

    lock (first)
    {
        lock (second)
        {
            from.Balance -= amount;
            to.Balance += amount;
        }
    }
}
```

### Стратегія 2: Monitor.TryEnter з Timeout

Усуваємо умову **No Preemption**: lock автоматично "відпускається" якщо очікування перевищило ліміт:

![Monitor.TryEnter flowchart](/images/system-programming/06/monitor-tryenter.png)

```csharp showLineNumbers [TryEnterDeadlock.cs]
using System.Threading;

public bool TryTransferFunds(Account from, Account to, decimal amount, int timeoutMs = 500)
{
    bool fromLocked = false, toLocked = false;

    try
    {
        Monitor.TryEnter(from, timeoutMs, ref fromLocked);
        if (!fromLocked) return false;  // не вдалось → спробуємо пізніше

        Monitor.TryEnter(to, timeoutMs, ref toLocked);
        if (!toLocked) return false;    // не вдалось → звільняємо from у finally

        // Обидва захоплені!
        from.Balance -= amount;
        to.Balance += amount;
        return true;
    }
    finally
    {
        // ЗАВЖДИ звільняємо порядку захопленим
        if (toLocked)   Monitor.Exit(to);
        if (fromLocked) Monitor.Exit(from);
    }
}
```

## Livelock та Starvation

### Livelock: Активне Безглуздя

**Livelock** — потоки **активно** виконуються (CPU зайнятий), але **не прогресують** — кожен з них постійно реагує на дії іншого, але ситуація не вирішується.

![Livelock vs Deadlock](/images/system-programming/06/livelock-vs-deadlock.png)

Аналогія: два ввічливих джентльмени у вузькому коридорі. Обидва поступаються — одночасно стрибають вліво, потім вправо, потім знову вліво. Оба "рухаються активно", але нікуди не йдуть.

```csharp showLineNumbers [LivelockDemo.cs]
// Ілюстрація livelock через "надмірно ввічливий" retry
public bool TryTransferFunds(Account from, Account to, decimal amount)
{
    while (true)
    {
        bool fromLocked = Monitor.TryEnter(from, 0);  // non-blocking
        if (!fromLocked) continue;  // не вдалось → retry ОДРАЗУ

        bool toLocked = Monitor.TryEnter(to, 0);
        if (!toLocked)
        {
            Monitor.Exit(from);  // звільняємо from і...
            continue;            // ...ОДРАЗУ ретраємо (наступний може зробити те саме)
        }

        try { from.Balance -= amount; to.Balance += amount; return true; }
        finally { Monitor.Exit(to); Monitor.Exit(from); }
    }
    // Якщо два потоки роблять це одночасно:
    // T1: lock from ✓, lock to ✗ → release from → retry
    // T2: lock to ✓, lock from ✗ → release to → retry
    // → обидва ніколи не отримають обидва locks → LIVELOCK
}

// ✅ Рішення livelock: додаємо випадкову паузу (jitter)
public bool TryTransferFundsSafe(Account from, Account to, decimal amount)
{
    var rand = new Random(Thread.CurrentThread.ManagedThreadId);

    for (int attempt = 0; attempt < 10; attempt++)
    {
        bool fromLocked = Monitor.TryEnter(from, 0);
        if (!fromLocked)
        {
            Thread.Sleep(rand.Next(10, 50));  // рандомна затримка → виходимо з livelock
            continue;
        }

        bool toLocked = Monitor.TryEnter(to, 0);
        if (!toLocked)
        {
            Monitor.Exit(from);
            Thread.Sleep(rand.Next(10, 50));  // рандомна затримка
            continue;
        }

        try { from.Balance -= amount; to.Balance += amount; return true; }
        finally { Monitor.Exit(to); Monitor.Exit(from); }
    }
    return false; // не вдалось за 10 спроб → результат з fallback логікою
}
```

### Starvation: Постійне Голодування

**Starvation** — окремий потік **ніколи** або дуже рідко отримує lock, хоча черга не є deadlocked. Інші потоки "перехоплюють" lock перед ним. Зазвичай пов'язана з пріоритетами або несправедливим планувальником.

```csharp showLineNumbers [StarvationDemo.cs]
// Сценарій: один "жирний" потік постійно тримає lock,
// не даючи "тонким" потокам виконатись
object _lock = new object();
long _starvingThreadExecutions = 0;

// 4 "агресивних" потоки — беруть lock часто і на довго
for (int i = 0; i < 4; i++)
{
    new Thread(() =>
    {
        while (true)
        {
            lock (_lock)
            {
                Thread.SpinWait(10_000);  // тримаємо lock довго
            }
        }
    }) { Priority = ThreadPriority.AboveNormal }.Start();
}

// 1 "голодуючий" потік
new Thread(() =>
{
    while (true)
    {
        lock (_lock)  // ← рідко отримує через постійну конкуренцію
        {
            Interlocked.Increment(ref _starvingThreadExecutions);
        }
    }
}) { Priority = ThreadPriority.Normal }.Start();

Thread.Sleep(5000);
Console.WriteLine($"Голодуючий потік виконався: {_starvingThreadExecutions} разів");
// Значно менше, ніж кожен з агресивних потоків!
```
