# LINQ-запити: від Where до GroupBy

## Що відбувається, коли ви пишете LINQ

Розберемо повільно, крок за кроком, що відбувається з цим рядком:

```csharp
var books = await context.Books
    .Where(b => b.Price > 100 && b.Year > 2010)
    .OrderBy(b => b.Title)
    .Select(b => new { b.Id, b.Title, b.Price })
    .ToListAsync();
```

**Крок 1 — `context.Books`:** повертає `DbSet<Book>`, що реалізує `IQueryable<Book>`. На цьому кроці не виконується жодного коду бази даних. `IQueryable` — це потенційний запит, що ще не виконався.

**Крок 2 — `.Where(b => b.Price > 100 && b.Year > 2010)`:** лямбда `b => b.Price > 100 && b.Year > 2010` **не виконується** як C#-код. Вона перетворюється на **Expression Tree** — структуру даних, що описує вираз у вигляді дерева вузлів: «властивість Price», «більше ніж», «константа 100», «і», «властивість Year», «більше ніж», «константа 2010».

**Крок 3 — `.OrderBy(b => b.Title)`:** додає новий вузол до Expression Tree: «сортувати за Title».

**Крок 4 — `.Select(b => new { b.Id, b.Title, b.Price })`:** додає проєкцію — «вибрати лише ці три поля».

**Крок 5 — `.ToListAsync()`:** це **термінальний оператор**. Він ініціює матеріалізацію: EF Core передає зібраний Expression Tree у **Query Compiler**, що аналізує його і транслює у SQL. SQL передається провайдеру, виконується на базі, результат повертається у вигляді рядків. EF Core матеріалізує рядки в C#-об'єкти.

Ось ключова думка: **LINQ-вирази над `IQueryable` не є C#-кодом, що виконується**. Вони є декларативним описом запиту, що перекладається у SQL. Це принципово відрізняється від LINQ над `IEnumerable`, де лямбда виконується у C#.

## IQueryable vs IEnumerable: критична різниця

Це одна з найважливіших концепцій для розуміння ефективності запитів:

```csharp
// IQueryable: ланцюжок Where, Select, OrderBy будується у SQL
// Фільтрація відбувається на рівні БД
IQueryable<Book> query = context.Books; // IQueryable
var cheapBooks = query.Where(b => b.Price < 100); // SQL де умова
var result = cheapBooks.Select(b => b.Title).ToListAsync(); // SELECT Title WHERE Price < 100

// IEnumerable: дані вже в пам'яті, LINQ виконується у C#
// Фільтрація відбувається в пам'яті ПІСЛЯ завантаження всіх даних
IEnumerable<Book> allBooks = await context.Books.ToListAsync(); // ← SELECT * завантажено!
var cheapBooksInMemory = allBooks.Where(b => b.Price < 100); // C# LINQ у пам'яті
```

Різниця є критичною: у першому варіанті база повертає лише дешеві книги. У другому — всі 100,000 книг завантажуються в пам'ять, і тільки потім C# відфільтровує їх.

::warning
Як тільки ви викликали `ToListAsync()`, `ToArrayAsync()`, `AsEnumerable()` або будь-який термінальний оператор — подальші LINQ-методи виконуються у C#, а не перекладаються у SQL. Це **найпоширеніша причина N+1, зайвого завантаження та проблем з пам'яттю** у EF Core-проєктах.
::

### Збереження IQueryable для композиції

IQueryable дозволяє **поступово будувати** запит і виконати його лише один раз:

```csharp
// Базовий запит без умов
IQueryable<Book> query = context.Books.AsQueryable();

// Умовне додавання фільтрів (без виконання до кінця)
if (!string.IsNullOrEmpty(titleFilter))
    query = query.Where(b => b.Title.Contains(titleFilter));

if (minPrice.HasValue)
    query = query.Where(b => b.Price >= minPrice.Value);

if (maxPrice.HasValue)
    query = query.Where(b => b.Price <= maxPrice.Value);

// Сортування залежить від параметра
query = sortBy switch
{
    "price_asc"  => query.OrderBy(b => b.Price),
    "price_desc" => query.OrderByDescending(b => b.Price),
    _            => query.OrderBy(b => b.Title)
};

// Пагінація
query = query.Skip((page - 1) * pageSize).Take(pageSize);

// Один SQL-запит з усіма умовами, сортуванням і пагінацією
var books = await query.ToListAsync();
```

Це потужний патерн — «динамічний запит». Один SQL виконується в кінці, але із усіма умовами, що накопичувались по шляху.

## Термінальні оператори: коли виконується SQL

Знати, які оператори ініціюють виконання SQL, — фундаментально для розуміння ефективності:

::field-group

::field{name="ToListAsync()" type="Termination"}
Найпоширеніший. Повертає `List<T>`. Завантажує всі рядки результату.
::

::field{name="ToArrayAsync()" type="Termination"}
Аналог ToListAsync, але повертає масив. Дещо швидше при великих результатах через відсутність resize-операцій List.
::

::field{name="FirstAsync() / FirstOrDefaultAsync()" type="Termination"}
Повертає перший елемент. `FirstAsync` кидає виключення якщо немає жодного, `FirstOrDefaultAsync` повертає null/default. SQL: `SELECT ... LIMIT 1`.
::

::field{name="SingleAsync() / SingleOrDefaultAsync()" type="Termination"}
Аналог First, але кидає виключення якщо знайдено більше одного елемента. SQL: `SELECT ... LIMIT 2` — щоб перевірити дублікати.
::

::field{name="FindAsync(keyValues)" type="Termination"}
Тільки для DbSet\<T\>. Спочатку перевіряє ChangeTracker — якщо сутність вже завантажена з таким PK, повертає її без SQL.
::

::field{name="CountAsync() / LongCountAsync()" type="Termination"}
SQL: `SELECT COUNT(*)`. Не завантажує рядки — тільки кількість.
::

::field{name="AnyAsync(condition)" type="Termination"}
SQL: `SELECT CASE WHEN EXISTS (...) THEN 1 ELSE 0 END`. Ефективніше за Count > 0.
::

::field{name="AllAsync(condition)" type="Termination"}
SQL: `SELECT ... WHERE NOT (condition)` — перевіряє чи виконується умова для всіх рядків.
::

::field{name="SumAsync() / AverageAsync() / MinAsync() / MaxAsync()" type="Termination"}
Агрегатні функції. SQL: `SELECT SUM(Price)`, `SELECT AVG(Price)` тощо.
::

::field{name="ExecuteDeleteAsync() / ExecuteUpdateAsync()" type="Termination"}
EF Core 7+. Виконує DELETE або UPDATE без завантаження сутностей у пам'ять.
::

::

## Проєкції Select: тільки потрібні дані

Проєкція через `Select` — одна з найважливіших практик оптимізації. Без неї EF Core виконує `SELECT *`, завантажуючи всі стовпці.

### Anonymous Types (анонімні типи)

```csharp
// Завантажуємо тільки те, що потрібно для відображення
var bookList = await context.Books
    .Select(b => new
    {
        b.Id,
        b.Title,
        b.Price,
        AuthorName = b.Author.Name // навігаційна властивість в проєкції → JOIN
    })
    .ToListAsync();
// SQL: SELECT b.Id, b.Title, b.Price, a.Name AS AuthorName
//      FROM Books b INNER JOIN Authors a ON b.AuthorId = a.Id
```

Зверніть: `b.Author.Name` у проєкції `Select` — EF Core автоматично виконає JOIN до таблиці `Authors`. Не потрібен явний `Include()`! Select з навігаційними властивостями і Include — різні механізми, і при проєкції Include зайвий.

### Typed Projections (DTO-класи)

Для передачі даних з шару даних у шар відображення краще використовувати конкретні DTO-класи замість анонімних типів:

```csharp
public record BookSummaryDto(int Id, string Title, decimal Price, string AuthorName);

var bookDtos = await context.Books
    .Select(b => new BookSummaryDto(
        b.Id,
        b.Title,
        b.Price,
        b.Author.Name))
    .ToListAsync();
```

`record` — ідеальний тип для DTO: immutable, value-based equality, лаконічний синтаксис.

### Вкладені проєкції

```csharp
public record AuthorWithBooksDto(
    int Id,
    string Name,
    IEnumerable<BookSummaryDto> Books);

var authorsWithBooks = await context.Authors
    .Select(a => new AuthorWithBooksDto(
        a.Id,
        a.Name,
        a.Books.Select(b => new BookSummaryDto(b.Id, b.Title, b.Price, a.Name))))
    .ToListAsync();
// SQL: один запит з підзапитом або JOIN — EF Core вирішує оптимально
```

::tip
Проєкції в `Select` виконуються **на рівні SQL**. EF Core транслює `b.Author.Name` у `JOIN` і `new AuthorWithBooksDto(...)` у правильні `SELECT`-стовпці. Жодного `SELECT *` і жодних зайвих об'єктів у пам'яті.
::

## Where та умовна фільтрація

`Where` є декларативним — EF Core транслює лямбду у `WHERE`-умову SQL.

### Підтримувані операції

```csharp
// Порівняння
.Where(b => b.Price > 100)          // WHERE Price > 100
.Where(b => b.Price >= 100)         // WHERE Price >= 100
.Where(b => b.Year == 2024)         // WHERE Year = 2024
.Where(b => b.Year != 2000)         // WHERE Year <> 2000

// Логічні оператори
.Where(b => b.Price > 100 && b.IsAvailable)  // WHERE Price > 100 AND IsAvailable = 1
.Where(b => b.Price > 500 || b.IsFeatured)  // WHERE Price > 500 OR IsFeatured = 1
.Where(b => !b.IsDeleted)                   // WHERE IsDeleted = 0

// Рядкові операції
.Where(b => b.Title.StartsWith("Кобзар"))    // WHERE Title LIKE 'Кобзар%'
.Where(b => b.Title.EndsWith("ець"))         // WHERE Title LIKE '%ець'
.Where(b => b.Title.Contains("поема"))       // WHERE Title LIKE '%поема%'
.Where(b => b.Title.ToLower() == "кобзар")  // WHERE LOWER(Title) = 'кобзар'
.Where(b => b.Title.Length > 10)            // WHERE LEN(Title) > 10 (SQL Server) / LENGTH(Title) > 10 (PG)

// Null перевірки
.Where(b => b.Description == null)          // WHERE Description IS NULL
.Where(b => b.Description != null)          // WHERE Description IS NOT NULL

// InRange / Contains (IN (...))
var validYears = new[] { 2020, 2021, 2022 };
.Where(b => validYears.Contains(b.Year))    // WHERE Year IN (2020, 2021, 2022)

// Колекції навігаційних властивостей
.Where(a => a.Books.Any())                  // WHERE EXISTS (SELECT 1 FROM Books WHERE AuthorId = a.Id)
.Where(a => a.Books.Any(b => b.Price > 200)) // WHERE EXISTS (SELECT 1 ... AND Price > 200)
.Where(a => a.Books.All(b => b.IsAvailable)) // WHERE NOT EXISTS (SELECT 1 ... AND NOT IsAvailable)
.Where(a => a.Books.Count() > 5)            // WHERE (SELECT COUNT(*) FROM Books WHERE AuthorId = a.Id) > 5
```

### EF.Functions: провайдерно-специфічні функції у Where

```csharp
// LIKE з підстановочними знаками (% та _)
.Where(b => EF.Functions.Like(b.Title, "%[0-9]%")) // SQL Server: LIKE

// ILike (PostgreSQL): регістронезалежний Like
.Where(b => EF.Functions.ILike(b.Title, "%роман%"))

// DateDiff (SQL Server)
.Where(b => EF.Functions.DateDiffDay(b.PublishedAt, DateTime.Now) < 30)

// Full text search (PostgreSQL)
.Where(a => EF.Functions.ToTsVector("ukrainian", a.Biography)
                         .Matches(EF.Functions.ToTsQuery("ukrainian", "поет")))
```

## OrderBy і сортування

```csharp
// Просте сортування
.OrderBy(b => b.Title)                   // ASC
.OrderByDescending(b => b.Price)         // DESC

// Багаторівневе сортування
.OrderBy(b => b.Author.LastName)         // первинне
    .ThenBy(b => b.Author.FirstName)     // вторинне
    .ThenByDescending(b => b.Year)       // третинне DESC

// Динамічне сортування за рядковим ім'ям поля
// Потребує бібліотеки System.Linq.Dynamic.Core:
.OrderBy("Price DESC, Title ASC")
```

::warning
Не виконуйте сортування **після** `ToListAsync()` — це сортування в пам'яті і не означає `ORDER BY` у SQL. Сортування для пагінації обов'язково має бути до матеріалізації.
::

## Join: явні і неявні

EF Core рідко вимагає явного `Join()` — коли є навігаційні властивості, EF Core виконає JOIN автоматично при `Include()` або в проєкції `Select`. Але іноді потрібен явний Join — наприклад, коли немає навігаційних властивостей або потрібен CROSS JOIN.

### Implicit Join через Select з навігацією (рекомендовано)

```csharp
// Немає явного Join — EF Core вирішує тип JOIN за навігацією
var data = await context.Books
    .Select(b => new
    {
        b.Title,
        b.Price,
        AuthorName = b.Author.Name,        // INNER JOIN (Author required)
        PublisherName = b.Publisher!.Name  // LEFT JOIN (Publisher optional)
    })
    .ToListAsync();
```

Навігаційна власність non-nullable → INNER JOIN. Nullable → LEFT JOIN.

### Explicit Join через метод Join()

```csharp
// Явний INNER JOIN коли навігаційних властивостей немає
var result = await context.Books
    .Join(
        context.Authors,          // до якої таблиці
        b => b.AuthorId,          // ключ з Books
        a => a.Id,                // ключ з Authors
        (b, a) => new             // проєкція результату
        {
            b.Title,
            b.Price,
            a.Name
        })
    .ToListAsync();
// SQL: SELECT b.Title, b.Price, a.Name FROM Books b INNER JOIN Authors a ON b.AuthorId = a.Id
```

### GroupJoin (LEFT JOIN)

```csharp
// GroupJoin + SelectMany = LEFT OUTER JOIN
var authorsWithBooks = await context.Authors
    .GroupJoin(
        context.Books,
        a => a.Id,
        b => b.AuthorId,
        (a, books) => new { Author = a, Books = books })
    .SelectMany(
        x => x.Books.DefaultIfEmpty(), // DefaultIfEmpty → LEFT JOIN
        (x, book) => new
        {
            x.Author.Name,
            BookTitle = book == null ? "(no books)" : book.Title
        })
    .ToListAsync();
// SQL: SELECT a.Name, b.Title FROM Authors a LEFT JOIN Books b ON a.Id = b.AuthorId
```

## GroupBy: агрегація на рівні SQL

`GroupBy` є складним для правильного використання в EF Core, бо не кожен `GroupBy`-запит може бути транслювальний у SQL. Розберемо детально.

### Коли GroupBy виконується у SQL

```csharp
// ✅ Простий GroupBy з агрегатами — виконується у SQL
var booksByYear = await context.Books
    .GroupBy(b => b.Year)
    .Select(g => new
    {
        Year       = g.Key,
        Count      = g.Count(),
        AvgPrice   = g.Average(b => b.Price),
        MinPrice   = g.Min(b => b.Price),
        MaxPrice   = g.Max(b => b.Price),
        TotalPrice = g.Sum(b => b.Price)
    })
    .OrderBy(x => x.Year)
    .ToListAsync();
// SQL: SELECT Year, COUNT(*), AVG(Price), MIN(Price), MAX(Price), SUM(Price)
//      FROM Books GROUP BY Year ORDER BY Year
```

```csharp
// ✅ GroupBy по навігаційній властивості
var booksByAuthor = await context.Books
    .GroupBy(b => new { b.AuthorId, b.Author.Name })
    .Select(g => new
    {
        AuthorId   = g.Key.AuthorId,
        AuthorName = g.Key.Name,
        BookCount  = g.Count(),
        AvgPrice   = g.Average(b => b.Price)
    })
    .OrderByDescending(x => x.BookCount)
    .ToListAsync();
// SQL: SELECT b.AuthorId, a.Name, COUNT(*), AVG(b.Price)
//      FROM Books b INNER JOIN Authors a ON b.AuthorId = a.Id
//      GROUP BY b.AuthorId, a.Name ORDER BY COUNT(*) DESC
```

### HAVING: фільтрація груп після GroupBy

```csharp
// ✅ Where після GroupBy → HAVING у SQL
var productiveAuthors = await context.Books
    .GroupBy(b => b.AuthorId)
    .Where(g => g.Count() >= 3)          // SQL: HAVING COUNT(*) >= 3
    .Select(g => new
    {
        AuthorId   = g.Key,
        BookCount  = g.Count(),
        TotalValue = g.Sum(b => b.Price)
    })
    .ToListAsync();
// SQL: SELECT AuthorId, COUNT(*), SUM(Price) FROM Books
//      GROUP BY AuthorId HAVING COUNT(*) >= 3
```

### Небезпечні GroupBy сценарії: Client Evaluation

Деякі GroupBy-запити EF Core не може транслювати у SQL і виконує в C#:

```csharp
// ⚠️ Якщо GroupBy не може бути переведений у SQL:
// В EF Core 3+ кидає InvalidOperationException (по дефолту)
// або виконує в C# ("client evaluation") — в залежності від конфігурації

// Цей запит повертає ВСІХ рядки і потім групує в пам'яті
var groups = await context.Books
    .AsEnumerable()          // ← Явно перемикаємось у C# LINQ!
    .GroupBy(b => b.Title[0]) // Перша літера назви — не підтримується у SQL
    .Select(g => new { Letter = g.Key, Count = g.Count() })
    .ToList();
// Краще: якщо потрібна групування по виразу, що не транслюється — raw SQL або Stored Procedure
```

## Підзапити та вкладені запити

EF Core підтримує підзапити через вкладені LINQ-вирази:

```csharp
// Повернути авторів, чия середня ціна книг вища за загальну середню
var avgPrice = await context.Books.AverageAsync(b => b.Price);

var premiumAuthors = await context.Authors
    .Where(a => a.Books.Average(b => b.Price) > avgPrice)
    .Select(a => new { a.Name, AvgBookPrice = a.Books.Average(b => b.Price) })
    .ToListAsync();
// SQL: SELECT a.Name, (SELECT AVG(Price) FROM Books WHERE AuthorId = a.Id) AS AvgBookPrice
//      FROM Authors a
//      WHERE (SELECT AVG(Price) FROM Books WHERE AuthorId = a.Id) > @avgPrice
```

Кореляційні підзапити:

```csharp
// Книги, ціна яких вища за середню вартість книг того самого автора
var aboveAverageBooks = await context.Books
    .Where(b => b.Price > context.Books
        .Where(other => other.AuthorId == b.AuthorId)
        .Average(other => other.Price))
    .Select(b => new { b.Title, b.Price })
    .ToListAsync();
// SQL: SELECT Title, Price FROM Books b
//      WHERE Price > (SELECT AVG(Price) FROM Books WHERE AuthorId = b.AuthorId)
```

## Пагінація: Skip і Take

Пагінація — базовий патерн для роботи зі списками. EF Core транслює `Skip().Take()` у `OFFSET/FETCH` (SQL Server), `LIMIT/OFFSET` (PostgreSQL/SQLite/MySQL):

```csharp
// Базова пагінація
int page = 2;
int pageSize = 20;

var booksPage = await context.Books
    .OrderBy(b => b.Id)       // Сортування ОБОВ'ЯЗКОВЕ для пагінації!
    .Skip((page - 1) * pageSize) // OFFSET 20
    .Take(pageSize)              // FETCH NEXT 20 ROWS ONLY
    .ToListAsync();
// SQL (SQL Server): SELECT ... FROM Books ORDER BY Id OFFSET 20 ROWS FETCH NEXT 20 ROWS ONLY

// Підрахунок загальної кількості для навігації
var totalCount = await context.Books.CountAsync();
var totalPages = (int)Math.Ceiling(totalCount / (double)pageSize);
```

::warning
**Обов'язкове сортування перед `Skip()`!** Без `OrderBy` SQL Server може кинути помилку, а інші провайдери можуть повертати різні рядки при різних виконаннях (недетермінований порядок рядків).
::

### Cursor-based пагінація (Keyset Pagination)

`OFFSET/LIMIT` має проблему при великих offset: `SKIP 100000` все одно змушує БД «прокрутити» 100,000 рядків. Краща альтернатива для великих даних — **keyset pagination** (пагінація за курсором):

```csharp
// Замість SKIP 100 → беремо рядки після відомого Id
int lastSeenId = 1000; // Id останнього рядка на попередній сторінці

var nextPage = await context.Books
    .Where(b => b.Id > lastSeenId) // ← cursor: не потрібен OFFSET!
    .OrderBy(b => b.Id)
    .Take(20)
    .ToListAsync();
// SQL: SELECT ... FROM Books WHERE Id > 1000 ORDER BY Id LIMIT 20
// Ефективно навіть на мільйонах рядків!
```

Keyset pagination ефективна для нескінченних прокруток (infinite scroll), але не підходить для «перейти на сторінку 5».

## Distinct, Union, Intersect, Except

```csharp
// DISTINCT
var uniqueYears = await context.Books
    .Select(b => b.Year)
    .Distinct()
    .OrderBy(y => y)
    .ToListAsync();
// SQL: SELECT DISTINCT Year FROM Books ORDER BY Year

// UNION (видаляє дублікати) та UNION ALL (залишає дублікати)
var allTitles = await context.Books
    .Select(b => b.Title)
    .Union(context.Articles.Select(a => a.Title)) // UNION
    .ToListAsync();

var allTitlesWithDuplicates = await context.Books
    .Select(b => b.Title)
    .Concat(context.Articles.Select(a => a.Title)) // UNION ALL
    .ToListAsync();

// INTERSECT: тільки спільні елементи
var commonTags = await context.BookTags
    .Select(bt => bt.TagId)
    .Intersect(context.ArticleTags.Select(at => at.TagId))
    .ToListAsync();

// EXCEPT: різниця множин (є у першому, немає у другому)
var booksWithoutOrders = await context.Books
    .Select(b => b.Id)
    .Except(context.OrderItems.Select(oi => oi.BookId))
    .ToListAsync();
```

## AsNoTracking: read-only запити

При виведенні даних, що не будуть змінюватись — завжди використовуйте `AsNoTracking()`. Це значно зменшує overhead Change Tracker'а:

```csharp
// Без Change Tracking — швидше для read-only
var books = await context.Books
    .AsNoTracking()
    .Where(b => b.IsAvailable)
    .Select(b => new BookDto(b.Id, b.Title, b.Price))
    .ToListAsync();
```

Що відбувається з `AsNoTracking()`:
- Об'єкти не реєструються у ChangeTracker (немає snapshot)
- Identity Map не перевіряється і не оновлюється
- Результат: ~20-30% менше пам'яті і ~15-25% більше швидкості для великих результатів

`AsNoTrackingWithIdentityResolution()` — аналог, але зберігає Identity Map для деduplication (якщо один автор є у кількох книгах — повертається один C#-об'єкт):

```csharp
var books = await context.Books
    .AsNoTrackingWithIdentityResolution()
    .Include(b => b.Author) // один Author-об'єкт, навіть якщо у 10 книгах
    .ToListAsync();
```

## Client vs Server Evaluation

EF Core прагне виконувати все у SQL (server evaluation). Але деякі вирази не можна перекласти у SQL — для них EF Core переключається на C# (client evaluation).

**Автоматичний Client Evaluation (тихий):**

```csharp
// Метод, що не має SQL-аналогу → EF Core завантажить всі рядки і виконає в C#
var books = await context.Books
    .Where(b => MyCustomFilter(b.Title)) // кастомна C#-функція
    .ToListAsync();
// EF Core завантажить ВСІ книги і потім відфільтрує в пам'яті!
```

**EF Core 3+: відмова від тихого client evaluation.** Починаючи з версії 3.0, EF Core кидає `InvalidOperationException` замість тихого client evaluation для `Where`-умов. Це правильне рішення — краще відразу отримати помилку, ніж несподіваний `SELECT *`.

Але для `Select` і `OrderBy` деякий client evaluation ще може відбуватись. Завжди перевіряйте SQL через логування.

**Явний перехід на client evaluation:**

```csharp
// Якщо вам справді потрібно виконати щось у C# після БД-запиту:
var books = await context.Books
    .Where(b => b.Price > 100) // БД-фільтр
    .AsEnumerable()            // ← перехід у C# LINQ
    .Where(b => MyComplexFilter(b)) // C# фільтр у пам'яті
    .ToList();
```

Але краще перенести таку логіку у `HasComputedColumnSql` або `HasDbFunction` (mapping кастомних SQL-функцій).

## Raw SQL та FromSqlRaw

Коли LINQ не вистачає або потрібна специфічна SQL-конструкція — EF Core дозволяє виконувати Raw SQL:

```csharp
// FromSqlRaw: SQL-рядок → результат матеріалізується у сутності
var books = await context.Books
    .FromSqlRaw("SELECT * FROM Books WHERE Price > {0} AND Year > {1}", 100, 2010)
    .OrderBy(b => b.Title) // можна продовжувати LINQ після FromSqlRaw!
    .ToListAsync();

// FromSqlInterpolated: безпечний через interpolation (параметризований)
var minPrice = 100;
var books2 = await context.Books
    .FromSqlInterpolated($"SELECT * FROM Books WHERE Price > {minPrice}")
    .ToListAsync();
// SQL: SELECT * FROM Books WHERE Price > @p0 ← параметр, не підстановка рядка!
```

::caution
Використовуйте `FromSqlInterpolated` замість `FromSqlRaw` для параметрів. `FromSqlRaw` з `string.Format` або string concatenation — **SQL INJECTION VULNERABILITY**. Завжди параметризуйте!
::

`ExecuteSqlRaw` для UPDATE/DELETE без повернення сутностей (але для масових операцій краще `ExecuteDeleteAsync` і `ExecuteUpdateAsync` з EF Core 7):

```csharp
// UPDATE без завантаження сутностей (EF Core 7+) — найкращий варіант
var updatedCount = await context.Books
    .Where(b => b.Price < 50)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(b => b.Price, b => b.Price * 1.1m)
        .SetProperty(b => b.LastModified, DateTime.UtcNow));

// DELETE без завантаження
var deletedCount = await context.Books
    .Where(b => !b.IsAvailable && b.Year < 1990)
    .ExecuteDeleteAsync();
```

## Антипатерни: запити, що вбивають продуктивність

::accordion

::accordion-item{label="N+1: завантаження у циклі" icon="i-lucide-triangle-alert"}
Найпоширеніший антипатерн. Один запит для списку сутностей + N запитів для кожної пов'язаної сутності окремо:

```csharp
// ❌ N+1: 1 запит для авторів + N запитів (один для кожного) для книг
var authors = await context.Authors.ToListAsync(); // 1 SQL
foreach (var author in authors)
{
    // EF Core (Lazy Loading) або NullReferenceException → ще 1 SQL для кожного!
    var bookCount = author.Books.Count;
    Console.WriteLine($"{author.Name}: {bookCount} books");
}

// ✅ Правильно: один запит з проєкцією
var authorStats = await context.Authors
    .Select(a => new { a.Name, BookCount = a.Books.Count() })
    .ToListAsync(); // Один SQL з підзапитом або JOIN
```
::

::accordion-item{label="Select * замість проєкції" icon="i-lucide-triangle-alert"}
```csharp
// ❌ Завантажуємо всі 30 стовпців для відображення 3
var orders = await context.Orders.ToListAsync();
var titles = orders.Select(o => o.OrderNumber).ToList();

// ✅ Завантажуємо тільки потрібне
var orderNumbers = await context.Orders
    .Select(o => o.OrderNumber)
    .ToListAsync();
```
::

::accordion-item{label="LINQ після AsEnumerable без потреби" icon="i-lucide-triangle-alert"}
```csharp
// ❌ AsEnumerable перемикає на C# LINQ — подальший Where у пам'яті
var result = context.Books
    .AsEnumerable()         // ← ВСІ книги завантажуються!
    .Where(b => b.Price > 100) // C# фільтр у пам'яті
    .ToList();

// ✅ Where до матеріалізації → SQL WHERE
var result2 = await context.Books
    .Where(b => b.Price > 100) // SQL WHERE
    .ToListAsync();
```
::

::accordion-item{label="Count() для перевірки існування" icon="i-lucide-triangle-alert"}
```csharp
// ❌ Count виконує SELECT COUNT(*) — зайва робота
if (context.Books.Where(b => b.AuthorId == id).Count() > 0) { ... }

// ✅ AnyAsync значно ефективніше: SELECT CASE WHEN EXISTS(...) THEN 1 ELSE 0 END
if (await context.Books.AnyAsync(b => b.AuthorId == id)) { ... }
```
::

::accordion-item{label="Пагінація без сортування" icon="i-lucide-triangle-alert"}
```csharp
// ❌ Skip без попереднього OrderBy — недетермінований порядок
var page = await context.Books.Skip(20).Take(10).ToListAsync();
// Може повертати різні рядки при кожному виконанні!

// ✅ Завжди сортуйте перед пагінацією
var page2 = await context.Books
    .OrderBy(b => b.Id)
    .Skip(20)
    .Take(10)
    .ToListAsync();
```

## Підсумок

::note
**Ключові думки цієї статті:**

- **LINQ над `IQueryable`** — декларативний опис запиту, що перекладається у SQL. Лямбди не виконуються як C#-код до матеріалізації.
- **`IQueryable` vs `IEnumerable`**: перше — SQL; друге — C# у пам'яті. Перехід між ними через `AsEnumerable()` або термінальний оператор.
- **Термінальні оператори** (`ToListAsync`, `FirstAsync`, `CountAsync` тощо) — момент коли виконується SQL.
- **`Select` для проєкцій**: завжди вибирайте тільки потрібні поля. Навігаційні властивості в `Select` → автоматичний JOIN без `Include`.
- **`GroupBy` з агрегатами** (Count, Sum, Avg etc.) транслюється у SQL GROUP BY. `Where` після `GroupBy` → `HAVING`.
- **`AsNoTracking()`** для read-only запитів: менше пам'яті, більше швидкості.
- **Client Evaluation**: EF Core 3+ кидає exception для не-трансльованих `Where`. Явний `AsEnumerable()` — усвідомлене рішення.
- **Пагінація**: `Skip().Take()` → `OFFSET/LIMIT`. Обов'язкове сортування. Для великих таблиць — Keyset Pagination.
- **Антипатерни**: N+1, SELECT *, Count() замість Any(), сортування після матеріалізації.
::

Наступна стаття — [Стратегії завантаження пов'язаних даних](/csharp/ef-core/loading-strategies) — детально розбирає Eager Loading (Include/ThenInclude), Lazy Loading (proxies і без), Explicit Loading, Select-проєкції, Split Queries і вплив кожної стратегії на SQL і продуктивність.
