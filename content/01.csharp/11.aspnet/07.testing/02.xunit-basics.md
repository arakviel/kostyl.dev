# xUnit — Основи модульного тестування в .NET

::note
Цей розділ є вашим першим практичним кроком у тестування. Ми напишемо перші реальні тести на C# і глибоко розберемо механіку фреймворку xUnit, який є стандартом де-факто для .NET проєктів.
::

## 1. Чому саме xUnit?

У світі .NET існують три основних тестових фреймворки: **xUnit.net**, **NUnit** та **MSTest**. Чому варто обирати xUnit?

::card-group

::card{title="xUnit.net" icon="i-lucide-check-circle"}

- Розроблений тими ж людьми, що і NUnit (але «з нуля», з виправленням помилок дизайну)
- Використовується командою .NET всередині Microsoft для тестування самого .NET 
- **Рекомендований** шаблоном `dotnet new` за замовчуванням
- Найчистіший API серед трьох: немає зайвих атрибутів, клас — це прихований `[TestClass]`

::

::card{title="NUnit" icon="i-lucide-minus-circle"}

- Ветеран (існує з 2002), дуже зрілий
- Популярний у legacy-проєктах
- Більш «багатослівний» синтаксис
- Все ще активно підтримується

::

::card{title="MSTest" icon="i-lucide-minus-circle"}

- Рідний Microsoft-фреймворк
- Інтегрований у Visual Studio «з коробки»
- Потребує атрибуту `[TestClass]` на кожному класі
- Менш популярний у open-source спільноті

::

::

У цьому курсі ми використовуємо **xUnit.net** як стандарт індустрії для нових .NET проєктів.

## 2. Налаштування тестового проєкту

## 3. Атрибут `[Fact]` — простий тест

`[Fact]` — це маркер для xUnit, що означає: «цей метод є тестом, запусти його». Метод не отримує параметрів ззовні.

Напишемо тести для простого класу-калькулятора. Спочатку — сам клас у основному проєкті:

```csharp [MyApp/Services/Calculator.cs]
namespace MyApp.Services;

public class Calculator
{
    public int Add(int a, int b) => a + b;
    public int Subtract(int a, int b) => a - b;
    public double Divide(int a, int b)
    {
        if (b == 0) throw new DivideByZeroException("Ділення на нуль заборонено.");
        return (double)a / b;
    }
}
```

Тепер — тести у тестовому проєкті:

```csharp [MyApp.Tests/Services/CalculatorTests.cs] showLineNumbers
using MyApp.Services;

namespace MyApp.Tests.Services;

public class CalculatorTests
{
    private readonly Calculator _sut; // SUT = System Under Test

    public CalculatorTests()
    {
        // xUnit створює новий екземпляр класу для КОЖНОГО тесту.
        // Конструктор — це наш "Arrange" для спільних даних.
        _sut = new Calculator();
    }

    [Fact]
    public void Add_GivenTwoPositiveNumbers_ReturnsCorrectSum()
    {
        // Arrange
        var a = 5;
        var b = 3;

        // Act
        var result = _sut.Add(a, b);

        // Assert
        Assert.Equal(8, result); // Очікуване першим, фактичне — другим
    }

    [Fact]
    public void Add_GivenNegativeNumbers_ReturnsCorrectSum()
    {
        // Arrange + Act (можна поєднати для простих випадків)
        var result = _sut.Add(-5, -3);

        // Assert
        Assert.Equal(-8, result);
    }

    [Fact]
    public void Divide_ByZero_ThrowsDivideByZeroException()
    {
        // Assert.Throws перевіряє, що код кидає виняток
        var exception = Assert.Throws<DivideByZeroException>(
            () => _sut.Divide(10, 0)
        );

        // Можна перевірити навіть повідомлення винятку
        Assert.Equal("Ділення на нуль заборонено.", exception.Message);
    }
}
```

**Анатомія цього коду:**

- **Рядки 7-13**: `_sut = new Calculator()` у конструкторі. xUnit принципово відрізняється від NUnit: він **не використовує `[SetUp]`**, натомість покладається на конструктор класу як ініціалізатор для кожного тесту. Це чистіший підхід.
- **Рядок 18**: `[Fact]` — маркер тесту.
- **Рядок 27**: `Assert.Equal(expected, actual)` — **зверніть увагу на порядок аргументів!** Спочатку очікуване значення, потім фактичне. Якщо переплутати, повідомлення про помилку буде заплутаним.
- **Рядки 36-45**: `Assert.Throws<T>()` — елегантний спосіб перевірити, що виняток виник.

::tip
**Золоте правило іменування**: `[MethodName]_[Scenario]_[ExpectedBehavior]`. Наприклад: `Divide_ByZero_ThrowsDivideByZeroException`. Така назва читається як специфікація поведінки і є самодокументуючою.
::

## 4. Атрибут `[Theory]` і `[InlineData]` — параметризовані тести

Уявіть, що вам потрібно перевірити метод `Add` для 10 різних пар чисел. Писати 10 окремих `[Fact]` тестів — це дублювання коду. Саме для цього існує `[Theory]`.

`[Theory]` позначає тест, який виконується **кілька разів з різними наборами вхідних даних**. Дані постачаються через атрибути `[InlineData]`, `[MemberData]` або `[ClassData]`.

```csharp [MyApp.Tests/Services/CalculatorTests.cs — розширення] showLineNumbers
// [Theory] + [InlineData]: запустить тест 5 разів з різними аргументами
[Theory]
[InlineData(2, 3, 5)]
[InlineData(-1, -1, -2)]
[InlineData(0, 0, 0)]
[InlineData(100, -100, 0)]
[InlineData(int.MaxValue - 1, 1, int.MaxValue)]
public void Add_VariousInputs_ReturnsCorrectSum(int a, int b, int expected)
{
    // Act
    var result = _sut.Add(a, b);

    // Assert
    Assert.Equal(expected, result);
}
```

Цей `[Theory]` генерує **5 окремих тестів** у Test Explorer. Якщо один провалиться — xUnit точно покаже, з якими аргументами це сталося.

Для складніших сценаріїв, де вхідні дані важко передати через `[InlineData]` (наприклад, об'єкти), використовуйте `[MemberData]`:

```csharp showLineNumbers
// Тестові дані оголошуємо статичним членом класу
public static IEnumerable<object[]> DivisionTestData => new List<object[]>
{
    new object[] { 10, 2, 5.0 },
    new object[] { 9, 3, 3.0 },
    new object[] { 7, 2, 3.5 },
};

[Theory]
[MemberData(nameof(DivisionTestData))]
public void Divide_VariousInputs_ReturnsCorrectQuotient(int a, int b, double expected)
{
    var result = _sut.Divide(a, b);
    Assert.Equal(expected, result);
}
```

## 5. Основні Assert-методи

xUnit надає багатий набір методів для перевірок. Ось найважливіші:

::field-group

::field{name="Assert.Equal(expected, actual)" type="Рівність"}
Найбазовий Assert. Перевіряє рівність двох значень. Для рядків — є чутливим до регістру. Є перевантаження для порівняння з точністю (`precision`) для `double`.
::

::field{name="Assert.NotEqual(expected, actual)" type="Нерівність"}
Протилежний до Equal. Переконується, що два значення НЕ рівні.
::

::field{name="Assert.True(condition) / Assert.False(condition)" type="Булеві вирази"}
Перевіряє, що умова є `true` або `false`. Використовуйте, коли немає більш специфічного Assert-методу.
::

::field{name="Assert.Null(object) / Assert.NotNull(object)" type="Null-перевірки"}
Перевіряє, що об'єкт є `null` або не є `null`. Незамінний для перевірки результатів пошуку.
::

::field{name="Assert.Throws<TException>(action)" type="Винятки"}
Перевіряє, що лямбда кидає виняток конкретного типу. Повертає сам об'єкт винятку для подальших перевірок.
::

::field{name="Assert.Contains(expected, collection)" type="Колекції"}
Перевіряє, що елемент міститься у колекції. Є аналог `Assert.DoesNotContain`.
::

::field{name="Assert.Empty(collection) / Assert.NotEmpty(collection)" type="Колекції"}
Перевіряє, що колекція порожня або непорожня.
::

::field{name="Assert.IsType<T>(object)" type="Типи"}
Перевіряє, що об'єкт є екземпляром вказаного типу. Корисно для перевірки типів повернення у поліморфних сценаріях.
::

::

## 6. Ізоляція тестів: важливий принцип xUnit

xUnit реалізує **ключовий принцип ізоляції**: для кожного тестового методу створюється **новий** екземпляр тестового класу. Це гарантує, що стан одного тесту не впливає на інший.

```csharp showLineNumbers
public class IsolationDemoTests
{
    private int _counter = 0; // Кожен тест отримує "свій" екземпляр з counter = 0

    [Fact]
    public void Test1_Increments_Counter()
    {
        _counter++;
        Assert.Equal(1, _counter); // Завжди 1, ніколи 2
    }

    [Fact]
    public void Test2_Also_Starts_From_Zero()
    {
        _counter++;
        Assert.Equal(1, _counter); // Також 1, бо це НОВИЙ екземпляр класу
    }
}
```

::caution
Порядок виконання тестів **не гарантований** і може змінюватися між запусками. Ніколи не покладайтеся на те, що один тест виконається до або після іншого. Якщо ваші тести залежать один від одного — це ознака поганого дизайну.
::

## 7. Пропускання тестів та виведення

Іноді тест треба тимчасово відключити або залишити коментар у виводі.

```csharp showLineNumbers
// Пропустити тест (буде позначено як "Skipped" у результатах)
[Fact(Skip = "Ця фіча ще не реалізована (Ticket #123)")]
public void FutureFeature_Test()
{
    // Цей код ніколи не виконається
    Assert.True(false);
}

// Додати виведення прямо з тесту (для дебагінгу)
public class TestWithOutput(ITestOutputHelper output)
{
    [Fact]
    public void TestWithLogging()
    {
        output.WriteLine("Починаємо тест...");
        var result = 2 + 2;
        output.WriteLine($"Результат: {result}");
        Assert.Equal(4, result);
    }
}
```

`ITestOutputHelper` — це ін'єктований xUnit-сервіс для виведення тексту. Використовуйте його замість `Console.WriteLine` (який у тестах часто не відображається).

## 9. Відмінності xUnit від NUnit (для тих, хто знайомий)

Якщо ви вже знайомі з NUnit або MSTest, ця таблиця допоможе зорієнтуватися:

| Поняття | xUnit | NUnit | MSTest |
|---|---|---|---|
| Маркер методу-тесту | `[Fact]` | `[Test]` | `[TestMethod]` |
| Маркер класу | *(не потрібен)* | `[TestFixture]` | `[TestClass]` |
| Параметризований тест | `[Theory]` + `[InlineData]` | `[TestCase]` | `[DataTestMethod]` + `[DataRow]` |
| Налаштування перед кожним тестом | Конструктор | `[SetUp]` | `[TestInitialize]` |
| Очищення після кожного тесту | `IDisposable.Dispose()` | `[TearDown]` | `[TestCleanup]` |
| Пропустити тест | `[Fact(Skip="")]` | `[Ignore("")]` | `[Ignore]` |

## 10. Резюме

Ми пройшли весь шлях від налаштування проєкту до написання реальних тестів:

- **`[Fact]`** — для тестів з фіксованими вхідними даними.
- **`[Theory]` + `[InlineData]`** — для параметризованих тестів (уникаємо дублювання коду).
- **Конструктор класу** — для ініціалізації SUT перед кожним тестом (замість `[SetUp]` у NUnit).
- **AAA-патерн** (Arrange-Act-Assert) — чітка структура кожного тесту.
- **Іменування** (`MethodName_Scenario_ExpectedBehavior`) — тест як специфікація поведінки.
