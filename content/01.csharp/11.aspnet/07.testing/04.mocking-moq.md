# Mocking з Moq — Ізоляція залежностей

## 1. Проблема: Тест із залежностями від зовнішнього світу

Уявіть: ви тестуєте `UserRegistrationService`, якому потрібен `IUserRepository` для перевірки існуючих email та збереження нових користувачів. Якщо використовувати справжній репозиторій із PostgreSQL:

- Тест потребує запущеної бази даних — **складно** налаштувати.
- Кожен тест читає/пише реальні дані — **тести впливають одне на одного**.
- Збереження в БД займає десятки мілісекунд — **тести стають повільними**.
- Якщо БД недоступна — **тест провалиться, хоча логіка правильна**.

Unit-тест повинен тестувати **виключно логіку** конкретного класу. Все зовнішнє — база даних, файлова система, HTTP-запити до сторонніх API — має бути замінено **тестовим дублером (test double)**.

Саме для цього існує **Moq** — найпопулярніша бібліотека для мокування в .NET.

## 2. Встановлення Moq

```bash
# У тестовому проєкті
dotnet add MyApp.Tests package Moq
```

Після встановлення `Moq` доступний у просторі імен `Moq`.

## 3. Базовий синтаксис: Створення Mock-об'єкта

Повернемося до нашого `UserRegistrationService`. Замість `FakeUserRepository` із попереднього розділу використаємо Moq:

```csharp [MyApp.Tests/Services/UserRegistrationServiceTests.cs] showLineNumbers
using Moq;
using MyApp.Services;

namespace MyApp.Tests.Services;

public class UserRegistrationServiceTests
{
    private readonly Mock<IUserRepository> _repositoryMock;
    private readonly UserRegistrationService _sut;

    public UserRegistrationServiceTests()
    {
        // Створюємо Mock-об'єкт для інтерфейсу
        _repositoryMock = new Mock<IUserRepository>();

        // .Object — це фактична реалізація інтерфейсу, яку ми передаємо в SUT
        _sut = new UserRegistrationService(_repositoryMock.Object);
    }
}
```

**Анатомія коду:**

- **Рядок 8**: `Mock<IUserRepository>` — Moq генерує клас, що реалізує `IUserRepository`, з усіма методами. За замовчуванням усі методи повертають значення за замовчуванням (`null`, `0`, `false`).
- **Рядок 14**: `_repositoryMock.Object` — властивість `.Object` повертає сам проксі-об'єкт, який можна передати в конструктор SUT.

## 4. Setup — Налаштування поведінки

Щоб Mock-об'єкт не просто «мовчав», а **відповідав конкретними значеннями**, використовується метод `.Setup()`.

```csharp showLineNumbers
[Fact]
public async Task RegisterAsync_WithExistingEmail_ThrowsException()
{
    // Arrange — налаштовуємо mock: коли кличуть ExistsAsync з будь-яким string, повертати true
    _repositoryMock
        .Setup(repo => repo.ExistsAsync(It.IsAny<string>()))
        .ReturnsAsync(true);

    // Act + Assert
    await Assert.ThrowsAsync<InvalidOperationException>(
        () => _sut.RegisterAsync("existing@test.com", "Pass123!")
    );
}

[Fact]
public async Task RegisterAsync_WithNewEmail_CallsSaveAsync()
{
    // Arrange — email не існує → ExistsAsync повертає false
    _repositoryMock
        .Setup(repo => repo.ExistsAsync("new@test.com"))
        .ReturnsAsync(false);

    // SaveAsync нічого не повертає (Task), тому Setup не потрібен — за замовчуванням Task.CompletedTask

    // Act
    await _sut.RegisterAsync("new@test.com", "Pass123!");

    // Assert — перевіримо нижче
}
```

### `It` — гнучкі матчери аргументів

`It` — це статичний клас Moq для гнучкого зіставлення аргументів у `Setup` та `Verify`:

::field-group

::field{name="It.IsAny<T>()" type="Matcher"}
Збігається з **будь-яким** аргументом типу `T`. Найчастіше вживаний матчер. Використовуйте, коли вам байдуже, яке конкретне значення передається.
::

::field{name="It.Is<T>(predicate)" type="Matcher"}
Збігається, якщо аргумент відповідає **умові-предикату**. Найпотужніший матчер для складних перевірок. Приклад: `It.Is<string>(s => s.Contains("@"))`.
::

::field{name="It.IsIn<T>(collection)" type="Matcher"}
Збігається, якщо аргумент є **одним зі значень** у колекції.
::

::field{name="It.IsNotNull<T>()" type="Matcher"}
Збігається з будь-яким ненульовим аргументом.
::

::

### Налаштування різних типів повернень

```csharp showLineNumbers
// Повернення значення
_mock.Setup(s => s.GetName()).Returns("John");

// Повернення Task<T> (async методи)
_mock.Setup(s => s.GetUserAsync(1)).ReturnsAsync(new User { Id = 1 });

// Повернення Task без значення (void async)
_mock.Setup(s => s.SaveAsync(It.IsAny<User>())).Returns(Task.CompletedTask);
// Або скорочено:
_mock.Setup(s => s.SaveAsync(It.IsAny<User>()));

// Кидання винятку
_mock.Setup(s => s.GetUserAsync(-1))
    .ThrowsAsync(new NotFoundException("User not found"));

// Різна поведінка при послідовних викликах
_mock.SetupSequence(s => s.GetCountAsync())
    .ReturnsAsync(0)   // перший виклик
    .ReturnsAsync(1)   // другий виклик
    .ReturnsAsync(2);  // третій виклик
```

## 5. Verify — Перевірка взаємодій

Основна відмінність Mock від Stub у тому, що Mock **відстежує всі виклики** та дозволяє перевіряти, чи відбувалися певні взаємодії та скільки разів.

```csharp showLineNumbers
[Fact]
public async Task RegisterAsync_WithNewEmail_SavesUserExactlyOnce()
{
    // Arrange
    _repositoryMock
        .Setup(repo => repo.ExistsAsync(It.IsAny<string>()))
        .ReturnsAsync(false);

    // Act
    await _sut.RegisterAsync("user@test.com", "Password123!");

    // Assert — перевіряємо, що SaveAsync був викликаний рівно 1 раз
    _repositoryMock.Verify(
        repo => repo.SaveAsync(It.IsAny<User>()),
        Times.Once
    );
}

[Fact]
public async Task RegisterAsync_WithExistingEmail_NeverCallsSaveAsync()
{
    // Arrange
    _repositoryMock
        .Setup(repo => repo.ExistsAsync(It.IsAny<string>()))
        .ReturnsAsync(true);

    // Act
    try { await _sut.RegisterAsync("existing@test.com", "Pass!"); } catch { }

    // Assert — SaveAsync не повинен бути викликаний ЖОДНОГО разу
    _repositoryMock.Verify(
        repo => repo.SaveAsync(It.IsAny<User>()),
        Times.Never
    );
}
```

### Times — маркери кількості

| Вираз | Значення |
|---|---|
| `Times.Once` | Рівно 1 раз |
| `Times.Never` | Жодного разу |
| `Times.AtLeastOnce` | Щонайменше 1 раз |
| `Times.Exactly(n)` | Рівно n разів |
| `Times.Between(min, max, Range.Inclusive)` | Від min до max разів (включно) |

## 6. Perевірка аргументів у Verify

Іноді важливо перевірити не лише факт виклику, а й **з якими аргументами**:

```csharp showLineNumbers
[Fact]
public async Task RegisterAsync_SavesUserWithCorrectEmail()
{
    // Arrange
    _repositoryMock
        .Setup(r => r.ExistsAsync(It.IsAny<string>()))
        .ReturnsAsync(false);

    const string email = "precise@test.com";

    // Act
    await _sut.RegisterAsync(email, "Pass123!");

    // Assert — перевіряємо, що SaveAsync викликаний з User, що має правильний email
    _repositoryMock.Verify(
        repo => repo.SaveAsync(
            It.Is<User>(u => u.Email == email) // Перевірка конкретного поля об'єкту
        ),
        Times.Once
    );
}
```

## 7. Комплексний приклад: Тестування `OrderService`

Зведемо всі концепції разом на складнішому прикладі. Уявімо `OrderService`, що залежить від двох сервісів:

```csharp [MyApp/Services/OrderService.cs] showLineNumbers
namespace MyApp.Services;

public interface IOrderRepository
{
    Task SaveAsync(Order order);
    Task<Order?> GetByIdAsync(int id);
}

public interface IEmailNotificationService
{
    Task SendOrderConfirmationAsync(string email, int orderId);
}

public record Order(int Id, string CustomerEmail, decimal Total, bool IsConfirmed = false);

public class OrderService(
    IOrderRepository orderRepository,
    IEmailNotificationService emailService)
{
    public async Task<int> PlaceOrderAsync(string customerEmail, decimal total)
    {
        if (total <= 0)
            throw new ArgumentException("Сума замовлення повинна бути додатною.");

        var order = new Order(
            Id: new Random().Next(1000, 9999),
            CustomerEmail: customerEmail,
            Total: total
        );

        await orderRepository.SaveAsync(order);
        await emailService.SendOrderConfirmationAsync(customerEmail, order.Id);

        return order.Id;
    }
}
```

Тести з повною ізоляцією обох залежностей:

```csharp [MyApp.Tests/Services/OrderServiceTests.cs] showLineNumbers
public class OrderServiceTests
{
    private readonly Mock<IOrderRepository> _orderRepoMock = new();
    private readonly Mock<IEmailNotificationService> _emailMock = new();
    private readonly OrderService _sut;

    public OrderServiceTests()
    {
        _sut = new OrderService(_orderRepoMock.Object, _emailMock.Object);
    }

    [Fact]
    public async Task PlaceOrderAsync_ValidOrder_SavesAndSendsEmail()
    {
        // Arrange — для void-async методів Setup необов'язковий
        // Moq за замовчуванням поверне Task.CompletedTask

        // Act
        var orderId = await _sut.PlaceOrderAsync("customer@shop.com", 250m);

        // Assert 1: повернутий ID у допустимому діапазоні
        Assert.InRange(orderId, 1000, 9999);

        // Assert 2: замовлення збережено рівно 1 раз
        _orderRepoMock.Verify(
            r => r.SaveAsync(It.Is<Order>(o =>
                o.CustomerEmail == "customer@shop.com" &&
                o.Total == 250m)),
            Times.Once
        );

        // Assert 3: email-підтвердження надіслано рівно 1 раз
        _emailMock.Verify(
            s => s.SendOrderConfirmationAsync("customer@shop.com", orderId),
            Times.Once
        );
    }

    [Theory]
    [InlineData(0)]
    [InlineData(-100)]
    public async Task PlaceOrderAsync_InvalidTotal_ThrowsArgumentException(decimal invalidTotal)
    {
        // Assert — виняток при некоректній сумі
        await Assert.ThrowsAsync<ArgumentException>(
            () => _sut.PlaceOrderAsync("customer@shop.com", invalidTotal)
        );

        // І нічого не збережено та не надіслано
        _orderRepoMock.Verify(r => r.SaveAsync(It.IsAny<Order>()), Times.Never);
        _emailMock.Verify(s => s.SendOrderConfirmationAsync(It.IsAny<string>(), It.IsAny<int>()), Times.Never);
    }
}
```

**Що важливо в цьому прикладі:**

- Рядки 6-7: **Скорочений синтаксис** `= new()` — C# 9 target-typed new expression. xUnit створить новий `Mock` для кожного тесту (через конструктор класу).
- Рядки 20-22: Для void/Task методів `Setup` можна опустити — Moq повертає `Task.CompletedTask` автоматично.
- Рядки 28-32: `It.Is<Order>(o => ...)` дозволяє **перевірити конкретні поля** переданого об'єкта, не покладаючись на рівність посилань.

## 8. MockBehavior: Strict vs Loose

За замовчуванням Moq використовує `MockBehavior.Loose` — будь-який виклик, на який немає `Setup`, повертає значення за замовчуванням. Це зручно, але може приховати проблеми.

```csharp showLineNumbers
// Loose (за замовчуванням) — незаконфігурований виклик поверне default(T)
var looseMock = new Mock<IUserRepository>();
// або явно:
var looseMock2 = new Mock<IUserRepository>(MockBehavior.Loose);

// Strict — будь-який незаконфігурований виклик КИДАЄ виняток
// Корисно для суворих тестів, де ви хочете знати про кожну взаємодію
var strictMock = new Mock<IUserRepository>(MockBehavior.Strict);
```

::tip
Починайте з `MockBehavior.Loose` для звичайних тестів. Використовуйте `Strict`, коли хочете гарантувати, що SUT не викликає жодних методів, крім тих, на які ви написали `Verify`.
::

## 9. Резюме

Moq — це потужний інструмент для ізоляції залежностей у unit-тестах:

- **`new Mock<IInterface>()`** → створити мок-об'єкт; **`.Object`** → отримати проксі для передачі.
- **`.Setup()`** → визначити поведінку при конкретному виклику.
- **`It.IsAny<T>()`**, **`It.Is<T>(pred)`** → гнучкі матчери аргументів.
- **`.Verify()`** + **`Times.*`** → перевірити, чи відбувалися очікувані взаємодії.
- **`MockBehavior.Strict`** → сувора перевірка всіх взаємодій.

У наступному розділі ми перейдемо від unit до **інтеграційного тестування** і навчимося піднімати реальний ASP.NET сервер у тестах.
