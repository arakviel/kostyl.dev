# Наслідування, MRO та суперсила `super()`

## Вступ: Навіщо потрібне наслідування та коли його використовувати

У парадигмі об'єктно-орієнтованого програмування **наслідування (Inheritance)** є механізмом, який дозволяє створити новий клас (дочірній, або підклас) на основі вже існуючого (базового, або суперкласу). Дочірній клас автоматично переймає атрибути та методи базового класу, що забезпечує повторне використання коду та побудову логічних ієрархій типів.

Розглянемо найпростіший приклад наслідування, де дочірній клас переймає функціонал батьківського класу без жодних змін:

```python
# simple_inheritance.py

class Vehicle:
    """Базовий клас для транспортних засобів."""
    def __init__(self, brand: str):
        self.brand = brand

    def start_engine(self) -> str:
        return f"Двигун транспортного засобу {self.brand} запущено."


class Car(Vehicle):
    """Дочірній клас, що наслідує Vehicle."""
    def honk(self) -> str:
        return "Бі-біп!"
```

Тут клас `Car` наслідує конструктор `__init__` та метод `start_engine` від класу `Vehicle`. Ми можемо переконатися в цьому, створивши об'єкт:

```python
my_car = Car("Toyota")

# Метод start_engine не описаний у Car, але доступний завдяки наслідуванню:
print(my_car.start_engine())  # Виведе: Двигун транспортного засобу Toyota запущено.

# Власний метод класу Car також працює:
print(my_car.honk())          # Виведе: Бі-біп!
```

Проте наслідування — це один із найсильніших зв'язків у розробці ПЗ (**tight coupling**). Зміна поведінки базового класу може неочікувано зламати логіку десятків його нащадків. Тому головне правило наслідування говорить: використовуйте його тільки тоді, коли між сутностями є чітке відношення **"is-a" (є об'єктом типу)**, а не **"has-a" (містить у собі)**. Для відношення "has-a" слід використовувати композицію або агрегацію.

## Одиночне наслідування та перевизначення методів (Method Overriding)

**Одиночне наслідування** передбачає наявність лише одного безпосереднього батьківського класу для підкласу.

У Python для наслідування базовий клас вказується в дужках після назви дочірнього класу. Якщо базовий клас не вказано явно, він автоматично наслідується від вбудованого класу `object`.

### Базовий синтаксис та перевизначення

**Перевизначення методів (Method Overriding)** — це створення у дочірньому класі методу з тим самим ім'ям, що й у батьківському класі. Це дозволяє дочірньому класу надати власну специфічну реалізацію для вже існуючої поведінки.

```python
# employee.py

class Employee:
    """Базовий клас для співробітників компанії."""
    def __init__(self, name: str, salary: float):
        self.name = name
        self.salary = salary

    def get_role_details(self) -> str:
        return f"Співробітник: {self.name}, Базовий оклад: {self.salary}"

    def calculate_bonus(self) -> float:
        # Стандартний бонус — 5% від зарплати
        return self.salary * 0.05


class Manager(Employee):
    """Дочірній клас для керівників."""
    def __init__(self, name: str, salary: float, team_size: int):
        # Ініціалізуємо базові атрибути через батьківський клас
        super().__init__(name, salary)
        self.team_size = team_size

    # Перевизначення методу get_role_details
    def get_role_details(self) -> str:
        # Додаємо специфічну для менеджера інформацію
        base_details = super().get_role_details()
        return f"{base_details}, Кількість підлеглих: {self.team_size}"

    # Перевизначення методу calculate_bonus
    def calculate_bonus(self) -> float:
        # Керівник отримує бонус 15% + додатково $100 за кожного підлеглого
        return (self.salary * 0.15) + (self.team_size * 100)
```

Давайте подивимося, як поводиться перевизначення методів на практиці:

```python
emp = Employee("Олена", 20000)
mgr = Manager("Олександр", 45000, 8)

print(emp.get_role_details())
# Виведе: Співробітник: Олена, Базовий оклад: 20000
print(f"Бонус Олени: {emp.calculate_bonus()}")
# Виведе: Бонус Олени: 1000.0

print(mgr.get_role_details())
# Виведе: Співробітник: Олександр, Базовий оклад: 45000, Кількість підлеглих: 8
print(f"Бонус Олександра: {mgr.calculate_bonus()}")
# Виведе: Бонус Олександра: 7550.0 (45000 * 0.15 + 8 * 100)
```

### Атрибути `__base__` та `__bases__`

Python зберігає інформацію про ієрархію наслідування безпосередньо в об'єктах класів. Для дослідження зв'язків використовуються спеціальні dunder-атрибути:

- `__base__`: Повертає перший безпосередній базовий клас.
- `__bases__`: Повертає кортеж усіх безпосередніх базових класів (корисно при множинному наслідуванні).

```python
print(Manager.__base__)
# Виведе: <class 'employee.Employee'>

print(Employee.__base__)
# Виведе: <class 'object'> (всі класи неявно наслідують object)

print(Manager.__bases__)
# Виведе: (<class 'employee.Employee'>,)
```

::note
Клас `object` є прабатьком усіх об'єктів та типом у Python. Якщо ви перевірите `object.__base__`, ви отримаєте `None`, оскільки він є вершиною ієрархії.
::

## Робота з `super()`: суперсила під капотом

Більшість програмістів вважають `super()` просто зручним способом виклику конструктора або методу батьківського класу. У найпростіших випадках одиночного наслідування це справді виглядає так. Проте під капотом Python ховає значно складніший і потужніший інструмент.

### Чому `super()` — це не просто «батьківський клас»?

Насправді `super()` повертає не безпосередньо батьківський клас, а **спеціальний проксі-об'єкт**, який динамічно перенаправляє пошук методів та атрибутів вздовж черги **MRO (Method Resolution Order)**.

Якщо ми використовуємо `super()` у методі класу, Python робить наступне:

1. Визначає поточний клас, у якому описано метод (через службове ім'я `__class__`).
2. Отримує поточний екземпляр класу (перший аргумент методу, зазвичай `self`).
3. Бере чергу MRO цього екземпляра (наприклад, `[Manager, Employee, object]`).
4. Знаходить у цій черзі поточний клас (`Manager`).
5. Починає пошук викликаного методу в MRO **одразу після** знайденого поточного класу (тобто в `Employee`, а потім в `object`).

Завдяки цьому `super()` гарантує, що у складних ієрархіях кожен батьківський метод буде викликано рівно один раз і у суворо визначеному порядку.

### Повна сигнатура `super()`

Зазвичай розробники пишуть `super()` без аргументів. Це синтаксичний цукор, доданий у Python 3. Насправді сигнатура функції виглядає так:

::math-formula
\\text{super(type, object\\\_or\\\_type)}
::

::field-group

::field{name="type" type="class"}
Клас, з якого починається відлік MRO. Пошук методів почнеться з наступного елемента після цього класу в черзі MRO.
::

::field{name="object_or_type" type="object | class"}
Об'єкт (екземпляр) або підклас, MRO якого використовується для пошуку.
::

::

Порівняємо використання:

::tabs

::tabs-item{label="З Python 3 (сучасний)"}

```python
class Manager(Employee):
    def __init__(self, name, salary, team_size):
        # Python автоматично підставляє super(Manager, self)
        super().__init__(name, salary)
```

::

::tabs-item{label="У стилі Python 2 (класичний)"}

```python
class Manager(Employee):
    def __init__(self, name, salary, team_size):
        # Доводилося явно вказувати клас та self
        super(Manager, self).__init__(name, salary)
```

::

::

Явне вказання параметрів у `super(Class, self)` досі є корисним у динамічних сценаріях, коли потрібно явно змінити точку входу в чергу MRO або викликати метод класу-предка через один рівень.

### Передача аргументів у `super().__init__` при кооперативному наслідуванні

Кови класи проектуються для роботи у складних ієрархіях (кооперативне наслідування), виникає проблема несумісності сигнатур конструкторів `__init__`. Кожен клас може приймати свій набір аргументів.

Якщо один з батьківських класів передасть у `super().__init__` несподівані аргументи, наступний клас у черзі MRO викине помилку `TypeError` (оскільки базовий клас `object.__init__` не приймає жодних аргументів).

Для розв'язання цієї проблеми використовується патерн `*args, **kwargs`:

```python
# cooperative.py

class GraphicElement:
    def __init__(self, color: str, *args, **kwargs):
        print("Ініціалізація GraphicElement")
        super().__init__(*args, **kwargs)  # Передаємо аргументи далі по MRO
        self.color = color


class ClickableElement:
    def __init__(self, action: str, *args, **kwargs):
        print("Ініціалізація ClickableElement")
        super().__init__(*args, **kwargs)  # Передаємо аргументи далі по MRO
        self.action = action


class Button(GraphicElement, ClickableElement):
    def __init__(self, color: str, action: str, label: str, *args, **kwargs):
        print("Ініціалізація Button")
        # Передаємо всі аргументи через super()
        super().__init__(color=color, action=action, *args, **kwargs)
        self.label = label
```

При створенні об'єкта `Button(color="red", action="click", label="Submit")` кожен конструктор ініціалізує свою частину об'єкта та передає решту аргументів далі по MRO, аж поки виклик не досягне `object.__init__`.

## Множинне наслідування та проблема «діаманта»

У попередніх версіях Python (до версії 2.2) використовувалися так звані **класи старого стилю (Classic Classes)**. Пошук методів здійснювався за простим алгоритмом **DFLS (Depth-First Search, Left-to-Right)** — спочатку вглиб, а потім зліва направо.

При алгоритмі DFLS черга пошуку для об'єкта класу `D` будувалася так:

::math-formula
\\text{D} \\rightarrow \\text{B} \\rightarrow \\text{A} \\rightarrow \\text{C} \\rightarrow \\text{A}
::

Після очищення дублікатів (залишався перший знайдений елемент) черга виглядала як: `[D, B, A, C]`.

**Чому це було катастрофою:**
Якщо клієнт викликав `d.speak()`, алгоритм шукав його в `D`, потім в `B`, а потім заходив углиб до пращура `A` і брав метод звідти. Метод, перевизначений у класі `C`, повністю ігнорувався! Це порушувало логіку наслідування: специфічні методи нащадків (клас `C`) повинні мати пріоритет над загальними методами далеких предків (клас `A`).

Ця проблема змусила розробників Python повністю переглянути механізм пошуку та створити **класи нового стилю**, які неявно наслідують `object`, та запровадити новий алгоритм — **C3-лінеаризацію**.

---

## Глибокий розбір алгоритму C3-лінеаризації та MRO

З приходом класів нового стилю у Python 2.2 (і повністю у Python 3) порядок пошуку методів регулюється алгоритмом **C3-лінеаризації**. Цей алгоритм будує плоский список класів — **MRO (Method Resolution Order)** — який визначає послідовність обходу ієрархії.

C3-лінеаризація гарантує дотримання трьох важливих властивостей:

1. **Локальний порядок предків (Local Precedence)**: Якщо клас `D` оголошений як `class D(B, C)`, то в MRO клас `B` завжди буде йти раніше за `C`.
2. **Монотонність (Monotonicity)**: Якщо клас `X` іде раніше за `Y` в MRO будь-котрого батьківського класу, то `X` повинен йти раніше за `Y` і в MRO дочірнього класу.
3. **Єдиність обходу**: Кожен клас відвідується один раз.

---

### Математична основа алгоритму C3

Формула лінеаризації класу :math-formula[C]{inline}, що наслідує класи :math-formula[B\_1, B\_2, \dots, B\_N]{inline}, записується так:

::math-formula
L[C(B\_1, B\_2, \dots, B\_N)] = [C] + \text{merge}(L[B\_1], L[B\_2], \dots, L[B\_N], [B\_1, B\_2, \dots, B\_N])
::

Де:

- :math-formula[L[C]]{inline} — лінеаризація (MRO) класу :math-formula[C]{inline}.
- :math-formula[[C]]{inline} — список, що містить один поточний клас.
- :math-formula[\text{merge}]{inline} — операція злиття кількох списків.

#### Правила роботи операції злиття (:math-formula[\text{merge}]{inline}):

1. Кожен список складається з **голови (Head)** — першого елемента списку, та **хвоста (Tail)** — решти елементів списку (може бути порожнім).
    > Наприклад, для списку `[A, B, C]`: голова — `A`, хвіст — `[B, C]`.
2. Починаємо з першого списку в операції :math-formula[\text{merge}]{inline}. Беремо його голову.
3. Перевіряємо, чи є ця голова у **хвостах** будь-яких інших списків, що беруть участь у злитті.
    - **Якщо її немає** у хвостах: ми вилучаємо її з усіх списків, додаємо до нашого результату MRO і переходимо до кроку 2 для пошуку наступного елемента.
    - **Якщо вона є** хоча б в одному хвості: ця голова заблокована. Ми переходимо до голови _наступного_ списку в :math-formula[\text{merge}]{inline} і робимо для неї таку ж перевірку.
4. Процес повторюється, поки всі класи не будуть вилучені. Якщо на якомусь кроці всі голови виявляються заблокованими, побудова MRO неможлива. Python зупиняє виконання та викидає помилку.

### Покроковий ручний розрахунок для «діаманта»

Давайте розрахуємо MRO для нашого попереднього прикладу:

```python
class A(object): pass
class B(A): pass
class C(A): pass
class D(B, C): pass
```

Розрахунок базових класів:

- :math-formula[L[\text{object}] = [\text{object}]]{inline}
- :math-formula[L[A] = [A] + \text{merge}(L[\text{object}], [\text{object}]) = [A, \text{object}]]{inline}
- :math-formula[L[B] = [B] + \text{merge}(L[A], [A]) = [B] + \text{merge}([A, \text{object}], [A]) = [B, A, \text{object}]]{inline}
- :math-formula[L[C] = [C, A, \text{object}]]{inline} (аналогічно до :math-formula[B]{inline})

Тепер розраховуємо :math-formula[L[D]]{inline}:

::math-formula
L[D] = [D] + \text{merge}(L[B], L[C], [B, C])
::

::math-formula
L[D] = [D] + \text{merge}([B, A, \text{object}], [C, A, \text{object}], [B, C])
::

1. **Крок 1**: Беремо першу голову — `B`.
    - Чи є `B` у хвості `[C, A, object]`? Ні (там його взагалі немає).
    - Чи є `B` у хвості `[B, C]`? Ні (у цьому списку `B` є головою, а не хвостом).
    - _Результат_: `B` вільний. Додаємо до MRO та видаляємо звідусіль.

    ::math-formula
    L[D] = [D, B] + \text{merge}([A, \text{object}], [C, A, \text{object}], [C])
    ::

2. **Крок 2**: Беремо першу голову — `A`.
    - Чи є `A` у хвості `[C, A, object]`? **Так** (у списку `[C, A, object]` елемент `A` знаходиться в хвості після `C`).
    - _Результат_: `A` заблокований. Переходимо до голови наступного списку — `C`.
    - Чи є `C` у хвості `[A, object]`? Ні.
    - Чи є `C` у хвості `[C]`? Ні (це єдиний елемент, тобто голова).
    - _Результат_: `C` вільний. Додаємо до MRO та видаляємо звідусіль.

    ::math-formula
    L[D] = [D, B, C] + \text{merge}([A, \text{object}], [A, \text{object}])
    ::

3. **Крок 3**: Беремо єдину голову — `A`.
    - Чи є `A` в якомусь хвості? Ні.
    - _Результат_: `A` вільний.

    ::math-formula
    L[D] = [D, B, C, A] + \text{merge}([\text{object}], [\text{object}])
    ::

4. **Крок 4**: Витягуємо `object`.
    - _Результат_: :math-formula[L[D] = [D, B, C, A, \text{object}]]{inline}.

Цей розрахунок доводить, що при виклику методу на об'єкті класу `D`, пошук піде по шляху `D -> B -> C -> A -> object`. Тобто перевизначений у `C` метод буде знайдено раніше, ніж застарілий у `A`!

### Конфлікт лінеаризації (Несумісний MRO)

Бувають випадки, коли розробник будує ієрархію, яка суперечить монотонності або локальному порядку предків. У такому разі Python видасть помилку на етапі компіляції класу.

Створимо такий конфлікт:

```python
# mro_conflict.py

class X: pass
class Y: pass

# A хоче, щоб X йшов перед Y
class A(X, Y): pass

# B хоче, щоб Y йшов перед X
class B(Y, X): pass

# C намагається наслідувати обидва класи
class C(A, B): pass
```

Спроба імпортувати або запустити цей код призведе до краху:

```text
TypeError: Cannot create a consistent method resolution order (MRO) for bases X, Y
```

Python захищає вас від створення суперечливої логіки, яку неможливо однозначно вирішити при пошуку методів.

---

### Налагодження MRO: `__mro__` та метод `mro()`

Для швидкої перевірки черги пошуку методів будь-якого класу використовуйте dunder-атрибут `__mro__` (повертає кортеж) або метод `mro()` (повертає список):

```python
print(D.mro())
# Виведе: [<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>]
```

## Патерн Mixins (Класи-домішки)

Коли ми будуємо архітектуру застосунку, множинне наслідування може стати джерелом помилок. Проте є один архітектурний патерн, який використовує множинне наслідування для вирішення складної задачі з мінімальними ризиками. Це — **Mixins (домішки)**.

**Mixin (клас-домішка)** — це невеликий клас, призначений виключно для додавання певної конкретної поведінки іншим класам. Домішки не використовуються для створення самостійних об'єктів.

### Ключові правила проектування Mixins

::card-group

::card{title="Немає стану" icon="i-heroicons-circle-stack"}
Домішка не повинна зберігати стан. У неї не повинно бути власного конструктора `__init__` (або, якщо він є, він має обов'язково викликати `super().__init__(*args, **kwargs)`).
::

::card{title="Single Responsibility" icon="i-heroicons-check-badge"}
Кожна домішка вирішує одну конкретну задачу: серіалізація, логування, авторизація, рендеринг тощо.
::

::card{title="Порядок у наслідуванні" icon="i-heroicons-arrows-right-left"}
У списку батьківських класів домішки завжди вказуються **перед** основним базовим класом (зліва направо), щоб їхні методи опинилися вище в MRO та могли перехопити виклики.
::

::

---

### Практична реалізація Mixins

Давайте реалізуємо два корисних Mixin-класи:

1. `JSONSerializableMixin` — додає об'єкту здатність серіалізуватися в JSON (ігноруючи приватні/захищені атрибути).
2. `LoggingMixin` — автоматично логує інформацію про створення та операції об'єкта.

```python
# mixins.py
import json

class JSONSerializableMixin:
    """Домішка для автоматичного перетворення об'єкта в JSON."""
    def to_json(self) -> str:
        # Відфільтровуємо приватні та захищені змінні (що починаються з _)
        public_data = {
            key: value
            for key, value in self.__dict__.items()
            if not key.startswith('_')
        }
        return json.dumps(public_data, ensure_ascii=False)


class LoggingMixin:
    """Домішка для логування операцій."""
    def log(self, message: str) -> None:
        # Використовуємо ім'я поточного класу динамічно
        class_name = self.__class__.__name__
        print(f"[LOG - {class_name}]: {message}")
```

Тепер об'єднаємо ці домішки в реальних класах бізнес-логіки. Зверніть увагу на порядок наслідування:

```python
# business_entities.py
from mixins import JSONSerializableMixin, LoggingMixin

class Product(JSONSerializableMixin, LoggingMixin):
    def __init__(self, name: str, price: float):
        self.name = name
        self.price = price
        # Використовуємо метод з LoggingMixin
        self.log(f"Створено продукт '{self.name}' за ціною ${self.price}")


class User(JSONSerializableMixin, LoggingMixin):
    def __init__(self, username: str, email: str):
        self.username = username
        self.email = email
        self._password_hash = "sha256$hashvalue"  # захищене поле
        self.log(f"Зареєстровано користувача {self.username}")
```

Перевіримо роботу нашого коду:

```python
prod = Product("Ноутбук", 35000.0)
# Виведе: [LOG - Product]: Створено продукт 'Ноутбук' за ціною $35000.0

usr = User("arakviel", "denys@example.com")
# Виведе: [LOG - User]: Зареєстровано користувача arakviel

# Виклик методу to_json з JSONSerializableMixin
print(prod.to_json())
# Виведе: {"name": "Ноутбук", "price": 35000.0}

print(usr.to_json())
# Виведе: {"username": "arakviel", "email": "denys@example.com"}
# Зверніть увагу: _password_hash автоматично відфільтровано!
```

Завдяки використанню домішок ми розширили функціональність класів `Product` та `User` без дублювання коду та без побудови складної вертикальної ієрархії типу `BaseEntity -> LoggingEntity -> SerializableEntity -> Product`.

## Практична лабораторія: Розробка системи транзакцій з аудитом та серіалізацією

Для практичного закріплення теми наслідування, MRO та домішок створимо модульну систему обробки банківських транзакцій. Наша мета — реалізувати класи транзакцій, які автоматично пишуть аудиторський лог та вміють конвертуватися в JSON для збереження в базу даних.

### Архітектура системи

Ми створимо:

1. **`AuditLogMixin`**: Домішка, що записує інформацію про кожну транзакцію в системний лог.
2. **`JSONSerializableMixin`**: Домішка для експорту даних у JSON (вже відома нам).
3. **`Transaction`**: Базовий клас транзакції, що містить ідентифікатор (`tx_id`) та суму (`amount`).
4. **`ValidatedTransaction`**: Клас, що наслідує `Transaction` та додає логіку перевірки лімітів (наприклад, сума транзакції не може бути меншою за $1 і більшою за $100,000).
5. **`SecureDepositTransaction`**: Кінцевий клас для поповнення рахунку, який поєднує логіку `ValidatedTransaction`, `AuditLogMixin` та `JSONSerializableMixin`.

Зверніть увагу, як ми використовуємо `*args` та `**kwargs` для передачі аргументів уздовж усього MRO ланцюжка.

### Покрокове вирішення та лістинг коду

::tabs

::tabs-item{label="Код модулів транзакцій"}

```python
# banking_system.py
import json
import uuid
from datetime import datetime

# 1. Домішки (Mixins)
class AuditLogMixin:
    def log_operation(self, status: str, details: str) -> None:
        timestamp = datetime.now().isoformat()
        class_name = self.__class__.__name__
        print(f"[{timestamp}] [AUDIT - {class_name}]: Status: {status} | Details: {details}")


class JSONSerializableMixin:
    def to_json(self) -> str:
        # Серіалізуємо словник атрибутів, ігноруючи приватні службові поля
        filtered_dict = {
            k: v for k, v in self.__dict__.items() if not k.startswith('_')
        }
        return json.dumps(filtered_dict, ensure_ascii=False)


# 2. Базовий клас транзакції
class Transaction:
    def __init__(self, amount: float, *args, **kwargs):
        print("-> Ініціалізація Transaction")
        super().__init__(*args, **kwargs)  # Передаємо залишок далі по MRO
        self.tx_id = str(uuid.uuid4())[:8]
        self.amount = float(amount)
        self.timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")

    def execute(self) -> None:
        print(f"Виконання базової транзакції {self.tx_id} на суму ${self.amount}")


# 3. Валідаційний клас з перевизначенням методу execute
class ValidatedTransaction(Transaction):
    def __init__(self, amount: float, max_limit: float = 100000.0, *args, **kwargs):
        print("-> Ініціалізація ValidatedTransaction")
        super().__init__(amount=amount, *args, **kwargs)
        self.max_limit = max_limit

    def execute(self) -> None:
        if self.amount < 1.0:
            raise ValueError("Сума транзакції не може бути меншою за $1.0")
        if self.amount > self.max_limit:
            raise ValueError(f"Перевищено максимальний ліміт транзакції у ${self.max_limit}")

        super().execute()  # Виклик наступного методу execute в MRO


# 4. Фінальний клас поповнення, що комбінує домішки та бізнес-класи
class SecureDepositTransaction(JSONSerializableMixin, AuditLogMixin, ValidatedTransaction):
    def __init__(self, amount: float, depositor_name: str, **kwargs):
        print("-> Ініціалізація SecureDepositTransaction")
        # Передаємо аргументи далі по MRO
        super().__init__(amount=amount, **kwargs)
        self.depositor_name = depositor_name

    def execute(self) -> None:
        self.log_operation("START", f"Початок депозиту для {self.depositor_name}")
        try:
            super().execute()  # Проходить валідацію та виконує транзакцію
            self.log_operation("SUCCESS", f"Депозит на суму ${self.amount} успішно проведено")
        except Exception as e:
            self.log_operation("FAILED", f"Помилка депозиту: {str(e)}")
            raise e
```

::card-group

::card{title="Композиція над наслідуванням" icon="i-heroicons-squares-plus"}
Уникайте глибоких ієрархій наслідування (більше 2-3 рівнів). Завжди надавайте перевагу **композиції** (створенню об'єкта як частини іншого об'єкта) над наслідуванням, якщо відношення не відповідає чіткому критерію "is-a".
::

::card{title="Кооперативний super()" icon="i-heroicons-arrow-path-solid"}
При множинному або кооперативному наслідуванні обов'язково використовуйте сигнатуру `*args, **kwargs` в `__init__` та викликайте `super().__init__`. Це запобігає виникненню помилок `TypeError` при обході MRO.
::

::card{title="Гігієна Mixins" icon="i-heroicons-shield-check"}
Класи-домішки (Mixins) мають бути невеликими, фокусуватися на одній задачі (SRP) та не містити внутрішнього стану. Домішки завжди вказуються ліворуч від основного базового класу.
::

::card{title="Перевірка MRO" icon="i-heroicons-magnifying-glass"}
Завжди перевіряйте чергу пошуку методів за допомогою `ClassName.mro()` або `ClassName.__mro__` у разі виникнення неочікуваної поведінки при множинному наслідуванні. Це найшвидший спосіб знайти помилки лінеаризації.
::
