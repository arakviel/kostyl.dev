# Deployment — декларативне управління репліками

## Проблема, яку вирішує Deployment

У попередній статті ми детально розглянули Pod і завершили важливим висновком: **Pod не слід створювати напряму у production**. Причина проста — Pod є ефемерним, не має self-healing, не підтримує масштабування та не дозволяє оновлювати застосунок без downtime.

Уявіть типовий сценарій: ви розгортаєте веб-API у Kubernetes. Вам потрібно:

1. **Три репліки** для балансування навантаження
2. **Автоматичне відновлення** — якщо одна репліка падає, система має створити нову
3. **Оновлення без downtime** — нова версія застосунку має розгортатись поступово
4. **Можливість rollback** — якщо нова версія має баг, потрібно швидко повернутись до попередньої

Якби ви використовували Pod напряму, довелося б:

```bash
# Створити 3 Pod вручну
kubectl apply -f api-pod-1.yaml
kubectl apply -f api-pod-2.yaml
kubectl apply -f api-pod-3.yaml

# Постійно моніторити їхній стан
kubectl get pods -w

# Якщо один Pod впав — вручну створити новий
kubectl apply -f api-pod-1.yaml

# Для оновлення — видалити старі та створити нові (з downtime!)
kubectl delete pod api-pod-1 api-pod-2 api-pod-3
kubectl apply -f api-pod-1-v2.yaml
kubectl apply -f api-pod-2-v2.yaml
kubectl apply -f api-pod-3-v2.yaml
```

Це неприйнятно для production. Саме тому існує **Deployment**.

::note
**Ключова ідея:** Deployment — це ресурс вищого рівня, який **автоматично** керує Pod. Ви описуєте бажаний стан (скільки реплік, яка версія образу), а Kubernetes сам створює, оновлює та відновлює Pod.
::

## Що таке Deployment

**Deployment** — це ресурс Kubernetes, який забезпечує декларативне управління набором ідентичних Pod. Він вирішує всі проблеми, які ми перелічили вище:

::card-group

::card{title="Self-healing" icon="i-heroicons-arrow-path"}
Якщо Pod видаляється або падає, Deployment автоматично створює новий, підтримуючи задану кількість реплік.
::

::card{title="Масштабування" icon="i-heroicons-arrows-pointing-out"}
Змінити кількість реплік можна однією командою або автоматично через HPA (Horizontal Pod Autoscaler).
::

::card{title="Rolling Updates" icon="i-heroicons-arrow-up-circle"}
Оновлення відбувається поступово: нові Pod створюються, перевіряються, і лише після цього старі видаляються. Завжди є працюючі репліки.
::

::card{title="Rollback" icon="i-heroicons-arrow-uturn-left"}
Kubernetes зберігає історію версій Deployment. Повернутись до попередньої версії можна однією командою.
::

::

### Аналогія з Docker Compose

У Docker Compose ви використовували `--scale` для створення кількох екземплярів:

```bash
docker compose up --scale api=3
```

Але це:
- Працює лише на одному хості
- Не відновлює контейнери автоматично при падінні хоста
- Не підтримує rolling updates

Deployment у Kubernetes — це `--scale` на стероїдах: працює на кластері, має self-healing та підтримує безпечні оновлення.

## Анатомія Deployment: структура маніфесту

Розглянемо базовий Deployment для веб-застосунку:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: app
        image: myapp:1.0.0
        ports:
        - containerPort: 8080
```

Розберемо структуру детально:

::field-group

::field{name="apiVersion" type="string" required="true"}
Для Deployment використовується `apps/v1` (не просто `v1`, як для Pod). Це означає, що Deployment належить до групи API `apps`.
::

::field{name="kind" type="string" required="true"}
Тип ресурсу — `Deployment`.
::

::field{name="metadata.name" type="string" required="true"}
Унікальне ім'я Deployment у межах namespace.
::

::field{name="spec.replicas" type="integer" default="1"}
Бажана кількість реплік Pod. Deployment Controller постійно підтримує саме цю кількість.
::

::field{name="spec.selector" type="object" required="true"}
Селектор для вибору Pod, якими керує цей Deployment. **Критично важливо:** мітки у `selector.matchLabels` мають збігатись з мітками у `template.metadata.labels`.
::

::field{name="spec.template" type="object" required="true"}
Шаблон Pod, який буде створюватись Deployment. Це повна специфікація Pod (без `apiVersion` та `kind`).
::

::

### Важливість selector та labels

Зверніть увагу на зв'язок між `selector` та `template.metadata.labels`:

```yaml
spec:
  selector:
    matchLabels:
      app: web      # ← Deployment шукає Pod з цією міткою
  template:
    metadata:
      labels:
        app: web    # ← Pod створюються з цією міткою
```

Якщо мітки не збігаються — Deployment не зможе знайти свої Pod і буде постійно створювати нові. Це типова помилка початківців.

::warning
**Типова помилка:** Змінити мітку у `template.metadata.labels`, але забути оновити `selector.matchLabels`. Deployment створить нові Pod, але старі не видалить, бо не зможе їх знайти.
::

## Створення першого Deployment

Створимо Deployment для простого веб-застосунку:

```yaml

kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
```

**Чому потрібен ReplicaSet?**

ReplicaSet відповідає за **підтримку заданої кількості реплік**. Його єдине завдання — гарантувати, що завжди працює рівно `N` Pod з певним шаблоном.

Deployment використовує ReplicaSet для реалізації **rolling updates**: при оновленні створюється новий ReplicaSet з новим шаблоном, а старий поступово зменшується до нуля.

Переглянути ReplicaSet:

```bash
kubectl get replicasets
```

::terminal-preview{title="kubectl get replicasets"}
<div class="line"><span class="opacity-40">$</span> <strong>kubectl get replicasets</strong></div>
<div class="line"></div>
<div class="line"><span class="text-gray-400">NAME                          DESIRED   CURRENT   READY   AGE</span></div>
<div class="line">nginx-deployment-7d6b8c9f4d   3         3         3       2m</div>
::

::note
**Важливо:** Ви **ніколи** не створюєте ReplicaSet напряму. Завжди використовуйте Deployment, який автоматично керує ReplicaSet. Пряме створення ReplicaSet позбавляє вас можливості rolling updates та rollback.
::

## Масштабування Deployment

Змінити кількість реплік можна трьома способами.

::

Тепер п'ять реплік. Kubernetes автоматично створив два нові Pod.

### Спосіб 2: Редагування маніфесту

Змініть `replicas` у YAML-файлі та застосуйте знову:

```yaml
spec:
  replicas: 2  # було 3
```

```bash
kubectl apply -f nginx-deployment.yaml
```

Kubernetes зменшить кількість реплік до 2, видаливши зайві Pod.

### Спосіб 3: Інтерактивне редагування

```bash
kubectl edit deployment nginx-deployment
```

Відкриється редактор з поточною конфігурацією Deployment. Змініть `replicas`, збережіть — зміни застосуються автоматично.

::tip
**Рекомендація:** У production завжди використовуйте **спосіб 2** (редагування маніфесту у Git). Це забезпечує версіонування та можливість code review. Команди `kubectl scale` та `kubectl edit` зручні для швидких тестів, але не для production.
::

## Rolling Updates — оновлення без downtime

Тепер розглянемо найпотужнішу можливість Deployment — **rolling update** (поступове оновлення).

### Сценарій: оновлення версії застосунку

Припустимо, ви випустили нову версію застосунку — `myapp:2.0.0`. Потрібно оновити всі репліки з `1.0.0` на `2.0.0` без downtime.

Змінимо образ у маніфесті:

```yaml
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: app
        image: myapp:2.0.0  # було 1.0.0
```

Застосуємо:

```bash
kubectl apply -f deployment.yaml
```

::terminal-preview{title="kubectl apply"}
<div class="line"><span class="opacity-40">$</span> <strong>kubectl apply -f deployment.yaml</strong></div>
<div class="line"></div>
<div class="line">deployment.apps/myapp-deployment <span class="text-blue-400 font-bold">configured</span></div>
::

```bash
kubectl get pods -w
```

2. Запущено перший новий Pod (`v2-p4k8t`)
3. Після того, як новий Pod став `Running` — видалено один старий Pod (`v1-8xk2p`)
4. Процес повторюється для решти Pod

**Ключовий момент:** У будь-який момент часу працює **принаймні 2 з 3 реплік**. Немає моменту, коли всі Pod недоступні. Це і є rolling update без downtime.

### Стратегії оновлення

Deployment підтримує дві стратегії оновлення:

::field-group

::field{name="RollingUpdate" type="стратегія" default="true"}
Поступове оновлення: нові Pod створюються, старі видаляються. Завжди є працюючі репліки. Використовується за замовчуванням.
::

::field{name="Recreate" type="стратегія"}
Спочатку видаляються всі старі Pod, потім створюються нові. Є момент downtime, але гарантується, що старі та нові версії не працюють одночасно.
::

::

Налаштування RollingUpdate:

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1  # Максимум 1 Pod може бути недоступним
      maxSurge: 1        # Максимум 1 додатковий Pod під час оновлення
```

::field-group

::field{name="maxUnavailable" type="integer | percentage" default="25%"}
Максимальна кількість Pod, які можуть бути недоступними під час оновлення. Може бути абсолютним числом (`1`) або відсотком (`25%`).
::

::field{name="maxSurge" type="integer | percentage" default="25%"}
Максимальна кількість додаткових Pod, які можуть бути створені понад `replicas` під час оновлення.
::

::

**Приклад:** Якщо `replicas: 10`, `maxUnavailable: 2`, `maxSurge: 2`:
- Під час оновлення може бути мінімум 8 працюючих Pod (`10 - 2`)
- Під час оновлення може бути максимум 12 Pod (`10 + 2`)

## Rollback — повернення до попередньої версії

Kubernetes зберігає історію Deployment. Якщо нова версія має баг — можна швидко повернутись до попередньої.

### Перегляд історії

```bash
kubectl rollout history deployment myapp-deployment
```

::terminal-preview{title="kubectl rollout history"}
<div class="line"><span class="opacity-40">$</span> <strong>kubectl rollout history deployment myapp-deployment</strong></div>
<div class="line"></div>
<div class="line"><span class="text-gray-400">REVISION  CHANGE-CAUSE</span></div>
<div class="line">1         &lt;none&gt;</div>
<div class="line">2         &lt;none&gt;</div>
<div class="line">3         &lt;none&gt;</div>
::

Кожна зміна у `spec.template` створює нову ревізію. За замовчуванням зберігається 10 останніх ревізій.

::tip
Щоб у історії відображалась причина зміни, додайте анотацію `kubernetes.io/change-cause`:

```bash
kubectl annotate deployment myapp-deployment kubernetes.io/change-cause="Update to version 2.0.0"
```
::

### Повернення до попередньої версії

```bash
kubectl rollout undo deployment myapp-deployment
```

Це повертає Deployment до попередньої ревізії (ревізія N-1).

Повернутись до конкретної ревізії:

```bash
kubectl rollout undo deployment myapp-deployment --to-revision=1
```

## Пауза та відновлення оновлення

Іноді потрібно зупинити rolling update (наприклад, для перевірки нової версії на частині реплік):

```bash
# Призупинити оновлення
kubectl rollout pause deployment myapp-deployment

# Перевірити нову версію вручну...

# Продовжити оновлення
kubectl rollout resume deployment myapp-deployment
```

Це дозволяє реалізувати **canary deployment** вручну: оновити частину реплік, перевірити метрики, і лише після цього оновити решту.

## Резюме

У цій статті ми розглянули Deployment — ключовий ресурс для управління Pod у production:

- **Deployment автоматизує управління Pod**: створює, оновлює, відновлює без вашого втручання
- **Self-healing**: якщо Pod падає — Deployment автоматично створює новий
- **Масштабування**: змінити кількість реплік можна однією командою (`kubectl scale`)
- **ReplicaSet**: проміжний шар між Deployment та Pod, який підтримує задану кількість реплік
- **Rolling Updates**: оновлення без downtime — нові Pod створюються, перевіряються, і лише після цього старі видаляються
- **Rollback**: історія версій дозволяє швидко повернутись до попередньої версії при проблемах
- **Стратегії оновлення**: `RollingUpdate` (за замовчуванням) та `Recreate`

Ключовий висновок: **Deployment — це стандартний спосіб розгортання stateless-застосунків у Kubernetes**. Ви описуєте бажаний стан (кількість реплік, версію образу), а система сама підтримує цей стан через reconciliation loop.

У наступній статті ми розглянемо **Service** — мережеву абстракцію, яка дозволяє звертатись до набору Pod через стабільну адресу, незалежно від того, скільки реплік працює і на яких вузлах вони розміщені.

## Практичні завдання

### Рівень 1 (Базовий)

**Завдання 1.** Створіть Deployment з іменем `web-app`, образом `nginx:1.25` та 3 репліками. Перевірте, що всі три Pod створені та працюють. Використайте `kubectl get deployments` та `kubectl get pods`.

**Завдання 2.** Видаліть один Pod з Deployment вручну через `kubectl delete pod`. Спостерігайте через `kubectl get pods -w`, як Deployment автоматично створює новий Pod. Поясніть, чому це відбувається.

**Завдання 3.** Збільште кількість реплік Deployment з 3 до 5 через команду `kubectl scale`. Перевірте, що Kubernetes створив два нові Pod. Потім зменште до 2 реплік та перевірте, що зайві Pod видалені.

### Рівень 2 (Практичний)

**Завдання 4.** Створіть Deployment з образом `nginx:1.24`. Оновіть образ на `nginx:1.25` через зміну маніфесту та `kubectl apply`. Спостерігайте за процесом rolling update через `kubectl get pods -w`. Поясніть, у якому порядку створюються нові Pod та видаляються старі.

**Завдання 5.** Створіть Deployment з `replicas: 4`, `maxUnavailable: 1`, `maxSurge: 2`. Виконайте оновлення образу. Під час оновлення підрахуйте максимальну та мінімальну кількість Pod, які працювали одночасно. Поясніть, як `maxUnavailable` та `maxSurge` впливають на процес.

**Завдання 6.** Створіть Deployment з образом `nginx:1.25`. Виконайте три оновлення (змінюйте образ на `1.24`, потім `1.23`, потім `1.22`). Використайте `kubectl rollout history` для перегляду історії. Поверніться до першої версії через `kubectl rollout undo --to-revision=1`.

### Рівень 3 (Дослідницький)

**Завдання 7.** Створіть Deployment з образом, який не існує (наприклад, `nginx:nonexistent`). Спостерігайте за поведінкою через `kubectl get pods` та `kubectl describe pod`. Поясніть, чому Deployment не створює нові Pod нескінченно, а зупиняється після кількох спроб.

**Завдання 8.** Створіть Deployment з 3 репліками. Виконайте `kubectl rollout pause` під час оновлення. Перевірте, що частина Pod оновилась, а частина залишилась на старій версії. Використайте `kubectl get replicasets` для перегляду двох ReplicaSet (старого та нового). Поясніть, як це можна використати для canary deployment.

**Завдання 9.** Створіть Deployment зі стратегією `Recreate` замість `RollingUpdate`. Виконайте оновлення образу та спостерігайте за процесом. Порівняйте з `RollingUpdate`: у чому різниця у поведінці? Коли `Recreate` може бути кращим вибором?

**Завдання 10.** Використайте `kubectl get deployment <назва> -o yaml` для перегляду повної специфікації Deployment. Знайдіть секцію `status` та поясніть значення полів: `replicas`, `updatedReplicas`, `readyReplicas`, `availableReplicas`. Створіть сценарій, коли ці значення будуть різними (наприклад, під час rolling update або коли Pod не проходить readiness probe).
