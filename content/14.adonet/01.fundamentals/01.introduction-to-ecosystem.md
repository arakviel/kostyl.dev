# Екосистема .NET Data Access: Навіщо нам ADO.NET у світі сучасних ORM?

## 1. Вступ та Контекст (Introduction & Context)

У сучасному світі розробки на платформі .NET для роботи з базами даних найчастіше використовується Entity Framework Core (EF Core) або інші Object-Relational Mapping (ORM) системи. Вони дозволяють нам працювати з базою даних так, ніби ми просто маніпулюємо звичайними колекціями об'єктів у пам'яті. Однак, за цією "магією" завжди ховається міцний, перевірений часом фундамент — **ADO.NET**.

### Hook: Історія двох розробників, гігабайти пам'яті та один нічний реліз

Уявіть собі реальну ситуацію з продуктового проєкту. Стартап стрімко зростає, і клієнти починають запитувати новий функціонал: експорт усієї історії транзакцій у формат CSV або Excel для їхніх бухгалтерів.

**Підхід розробника А (Магія ORM):**
Він знає EF Core, як свої п'ять пальців. Рішення здається елементарним і займає три рядки коду:

::code-group

```csharp [TransactionExportService.cs]
public async Task<byte[]> ExportAllTransactionsAsync(int customerId)
{
    // EF Core будує запит і витягує всі дані...
    List<Transaction> allTransactions = await _dbContext.Transactions
        .Where(t => t.CustomerId == customerId)
        .ToListAsync(); // [!code highlight]

    // Перетворення об'єктів у CSV масив байтів
    return _csvService.Generate(allTransactions);
}
```

::

::caution
**Що пішло не так?**
У перші місяці це працювало прекрасно (мало даних). Але коли у великого Enterprise-клієнта накопичилося 4 мільйони транзакцій, виклик цього методу змусив EF Core:

1. Витягнути всі 4 мільйони рядків з БД в пам'ять.
2. Виділити пам'ять під 4 мільйони C# об'єктів (Tracking `EntityEntry` для кожного з них).
3. Викликати збірку сміття (Garbage Collection), зупиняючи весь сервер.

Результат? `System.OutOfMemoryException` і "мертвий" бекенд, змушуючи всіх інших користувачів системи бачити сторінку помилки 502 Bad Gateway.
::

**Підхід розробника Б (ADO.NET Streaming):**
Розробник Б розуміє, що ORM не створений для потокової обробки великих обсягів даних. Він відкриває документацію і створює рішення на базі "голої" інфраструктури **ADO.NET**:

::code-collapse{title="Показати рішення на ADO.NET"}

```csharp [OptimizedExportService.cs]
public async Task StreamTransactionsToCsvAsync(int customerId, Stream outputStream)
{
    // 1. Отримуємо звичайний текстовий рядок підключення
    string connectionString = _configuration.GetConnectionString("Default");

    // 2. Створюємо та відкриваємо підключення (using гарантує звільнення пам'яті)
    await using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // 3. Формуємо SQL команду із параметром (захист від ін'єкцій)
    await using var command = new SqlCommand(
        "SELECT Date, Amount, Description FROM Transactions WHERE CustomerId = @custId",
        connection);

    command.Parameters.AddWithValue("@custId", customerId);

    // 4. Найголовніша магія - Отримуємо DataReader!
    await using SqlDataReader reader = await command.ExecuteReaderAsync();

    // 5. Записуємо заголовок CSV
    using var writer = new StreamWriter(outputStream);
    await writer.WriteLineAsync("Date,Amount,Description");

    // 6. Читаємо потоково по одному рядку за раз (!)
    while (await reader.ReadAsync()) // [!code focus]
    {
        // Дані живуть у пам'яті мілісекунду, ми їх одразу пишемо в потік
        var date = reader.GetDateTime(0).ToString("yyyy-MM-dd");
        var amount = reader.GetDecimal(1);
        var description = reader.GetString(2);

        await writer.WriteLineAsync($"{date},{amount},\"{description}\"");
    }

    await writer.FlushAsync();
}
```

::

#### Анатомія "Магії" DataReader-а

Давайте розберемо, що робить цикл у рядку 26 (`while (await reader.ReadAsync())`) таким унікальним:

1. **Контекст**: Ми маємо мільйони рядків у базі даних, і нам потрібно їх віддати користувачу через REST API у вигляді завантажуваного файлу.
2. **Код**: `SqlDataReader` не завантажує всю таблицю в пам'ять. Це — "пожежний шланг", підключений напряму до бази даних.
3. **Декомпозиція потоковості (Streaming)**:
    - База даних (SQL Server) знаходить рядки.
    - Вона пакує їх у невеличкі мережеві пакети (по протоколу TDS - Tabular Data Stream).
    - Ваш додаток (C# код) отримує один рядок, витягує з нього 3 поля (`GetDateTime`, `GetDecimal`, `GetString`).
    - Далі код негайно пише ці дані у вихідний мережевий потік до браузера користувача (`StreamWriter`).
    - Рядок "зникає" (стає доступним для Garbage Collector'a), і ми переходимо до наступного.

Таким чином, експорт **4 мільйонів рядків** використовує лише **кілька мегабайт оперативної пам'яті**!

### Що ми отримаємо в результаті цього курсу?

Вивчаючи ADO.NET, ви перестанете сприймати базу даних як "чорний ящик", що обробляється "якимось `DbContext`".
Ви зрозумієте:

- Як працюють підключення та чому виникають тайм-аути.
- Як безпечно передавати параметри і не стати жертвою SQL Injection.
- Як написати власний мікро-ORM, якщо вам здається, що існуючі рішення працюють неефективно.
- Чому виклики бази є найвужчим місцем (bottleneck) системи, і як асинхронність (`Task`) рятує продуктивність.

### Prerequisites (Попередні вимоги)

Щоб успішно засвоїти матеріал, переконайтеся, що ви знайомі з:

- Основами **C#** (Класи, об'єкти, методи, винятки, `using` директиви для `IDisposable`).
- Базовим **SQL** (Що таке таблиця, `SELECT`, `INSERT`, `UPDATE`, `WHERE`).
- Розумінням, як встановлювати **NuGet** пакети у ваш проєкт.

## 2. Історична Еволюція (Historical Evolution)

Технології доступу до даних не виникли з нізвідки. Щоби зрозуміти ADO.NET, потрібно зазирнути в історію розробки під Windows.

::mermaid
timeline
title Еволюція Data Access у світі Microsoft
1992 : ODBC
: (Open Database Connectivity) - Універсальний API на мові C для доступу до будь-якої РУБД (Реляційної Бази). Дуже швидкий, але складний у застосуванні.
1996 : OLE DB & ADO
: (ActiveX Data Objects) - COM-інтерфейси для доступу до реляційних та НЕреляційних даних (наприклад, поштові скриньки, Excel файли).
2002 : ADO.NET
: З виходом .NET Framework 1.0. Цілком нова архітектура. "Відключена" модель із `DataSet`. Native managed-провайдери, що розмовляють напряму з базою (напр. TDS для SQL Server).
2008 : Entity Framework
: Перша повноцінна спроба Microsoft створити ORM поверх ADO.NET. Спочатку дуже громіздка.
2011 : Dapper
: StackOverflow випускає свій Micro-ORM як тонкий надбудову над ADO.NET для максимальної продуктивності.
2016 : EF Core
: Повне переписування Entity Framework з нуля. Модульний, швидкий та кросплатформний, але "під капотом" все ще покладається на інтерфейси ADO.NET.
::

> **Головний висновок історії**: Кожна нова "абстракція" не замінювала попередню технічно, вона просто її **обгортала**. Робота з EF Core або Dapper сьогодні — це на 95% робота з тим самим ADO.NET, тільки прихованим за "синтаксичним цукром" (syntactic sugar).

## 3. Фундаментальні Концепції (Fundamental Concepts)

Сутність ADO.NET — забезпечити стандартизований, але при цьому високопродуктивний "міст" між вашою програмою та джерелом даних.

### Моделі роботи з даними (Connected vs. Disconnected)

Свого часу справжньою революцією став поділ способів обробки даних на дві парадигми: **Connected** (Підключена) та **Disconnected** (Відключена).

::tabs
::tabs-item{label="Connected Model" icon="i-lucide-plug"}
**Підключена модель (Connected Model)**

Цільові класи: `IDbConnection`, `IDbCommand`, `IDataReader`.

**Сутність**: Програма постійно утримує відкрите мережеве підключення до бази під час обробки.

- Ми відправляємо запит.
- Ми читаємо результати "на льоту".
- **Плюс**: Максимальна швидкодія, мінімальні витрати пам'яті програми. Це найкращий спосіб для читання (великих) об'ємів даних.
- **Мінус**: Поки ми обробляємо дані рядок за рядком (наприклад, виконуємо повільні розрахунки або чекаємо вводу від користувача), інфраструктура з'єднання _заблокована_. Сервер бази даних витрачає ресурси на підтримку сесії. Потрібно закривати підключення якнайшвидше!

::

::tabs-item{label="Disconnected Model" icon="i-lucide-unplug"}
**Відключена модель (Disconnected Model)**

Цільові класи: `DataSet`, `DataTable`, `DataAdapter`.

**Сутність**: Програма копіює шматок бази даних у свою оперативну пам'ять як локальний кеш.

- Ми відкриваємо з'єднання -> Завантажуємо результати запиту у локальний `DataSet` -> **Одразу закриваємо з'єднання**.
- Далі працюємо з даними у пам'яті (додаємо, видаляємо, змінюємо).
- Потім відкриваємо з'єднання знову і відправляємо "дельта-зміни" назад у базу.
- **Плюс**: Підключення звільняється миттєво. Легко перекидати дані між сервісами або передавати у UI.
- **Мінус**: Вся таблиця живе в RAM (Оперативній пам'яті). Це вб'є застосунок, якщо даних багато.

::note
_Історична довідка_: Disconnected модель із `DataSet/DataTable` була "зіркою" у 2000-х (епоха Windows Forms). Сьогодні, в епоху Web API та мікросервісів об'єкти `DataTable` майже не використовуються. Сучасна "Disconnected" модель — це коли ви завантажуєте дані у ваші класи-моделі (Records / POCO) через EF Core або Dapper, а `DataTable` залиште для роботи із Legacy-кодом.
::
::
::

### Прошарки та Провайдери Даних (Data Providers)

ADO.NET спирається на патерн "Абстрактна фабрика" (Abstract Factory) та загальні інтерфейси.
Вам не потрібно знати низькорівневі протоколи спілкування з базою. Кожна база має свій спеціальний **Data Provider** — бібліотеку-драйвер, що імплементує інтерфейси з `System.Data`.

::card-group

::card{title="SQL Server" icon="i-simple-icons-microsoftsqlserver"}
**Провайдер**: `Microsoft.Data.SqlClient`
_(Раніше відомий як System.Data.SqlClient)_.
Керує підключеннями через протокол **TDS** (Tabular Data Stream). Офіційно підтримується Microsoft, оптимізований для Azure SQL.
::

::card{title="PostgreSQL" icon="i-simple-icons-postgresql"}
**Провайдер**: `Npgsql`
Чудовий open-source провайдер. Не розробляється розробниками самої СУБД, але має бездоганну підтримку усіх новітніх фіч як-то JSON-B колонки чи масиви.
::

::card{title="MySQL / MariaDB" icon="i-simple-icons-mysql"}
**Провайдер**: `MySqlConnector` або офіційний `MySql.Data`.
_(Примітка: в індустрії зазвичай віддають перевагу MySqlConnector через його кращу продуктивність та реальну асинхронність під капотом)._
::

### Управління ресурсами: чому це життєво важливо?

Підключення до бази даних (`IDbConnection`) — це не просто об'єкт у пам'яті. Це:

1. TCP-сокет.
2. Сесія (потік автентифікації, перевірка паролів).
3. Робочий потік на боці самої СУБД, призначений персонально для вас, який "їсть" оперативку сервера бази.

Коли ви створюєте звичайну змінну у C# (напр. `string text = "hello"`), вам байдуже, що з нею буде далі — прийде Garbage Collector (Збирач Сміття) і видалить її.
Але Garbage Collector **поняття не має**, як закривати TCP/IP підключення у мережі. Якщо ви перестанете використовувати об'єкт `SqlConnection` і не сповістите його про закінчення роботи, з'єднання "зависне". Якщо таких з'єднань збереться 100 — пул вичерпається, і наступний користувач побачить помилку таймауту.

**Паттерн `IDisposable`**
Усі об'єкти ADO.NET (Connections, Commands, Readers) імплементують інтерфейс `IDisposable`. Тобто, мають метод `Dispose()`, який примусово і акуратно розриває зв'язки з зовнішнім світом операційної системи.
У C# для гарантованого виклику `Dispose()` існує ключове слово `using`.

## 5. Типологія та Порівняння (Typology & Comparison)

Чому ми не пишемо сьогодні ВСЕ на ADO.NET, якщо він такий класний і швидкий? Тому що написання коду на ADO.NET вимагає написання величезної кількості шаблонного (boilerplate) коду.

Давайте порівняємо, як виглядає отримання одного користувача по ID за допомогою трьох найкращих підходів:

::code-group

```csharp [1. Pure ADO.NET (Максимальний контроль)]
// ШВИДКОДІЯ: Найшвидший
// КОНТРОЛЬ: Максимальний
// ЗАТРАТИ ЧАСУ НА РОЗРОБКУ: Величезні

public async Task<User?> GetUserByIdAsync(int id)
{
    await using var conn = new SqlConnection(_conString);
    await using var cmd = new SqlCommand("SELECT Id, Name, Age FROM Users WHERE Id = @id", conn);
    cmd.Parameters.AddWithValue("@id", id);

    await conn.OpenAsync();
    await using var reader = await cmd.ExecuteReaderAsync();

    if (await reader.ReadAsync())
    {
        return new User
        {
            Id = reader.GetInt32(0),
            Name = reader.GetString(1), // А якщо воно NULL у базі? Впаде Exception (!). Треба перевіряти IsDBNull.
            Age = reader.IsDBNull(2) ? null : reader.GetInt32(2)
        };
    }
    return null;
}
```

```csharp [2. Dapper (Золота середина)]
// ШВИДКОДІЯ: Майже така сама як ADO.NET (відставання на 1-2%)
// КОНТРОЛЬ: Високий (Ми все ще самі пишемо SQL)
// ЗАТРАТИ ЧАСУ НА РОЗРОБКУ: Мінімальні

public async Task<User?> GetUserByIdAsync(int id)
{
    await using var conn = new SqlConnection(_conString);

    // Dapper розширює IDbConnection методами-розширеннями.
    // Він сам створює SqlCommand, передає параметри, читає DataReader
    // і через Reflection автоматично розподіляє дані у властивості класу User.
    return await conn.QueryFirstOrDefaultAsync<User>(
        "SELECT * FROM Users WHERE Id = @Id",
        new { Id = id }); // [!code highlight]
}
```

```csharp [3. EF Core (Ентерпрайз масштаб)]
// ШВИДКОДІЯ: Добра (але генерує запити з overhead'ом, повільніше за Dapper)
// КОНТРОЛЬ: Низький (SQL генерується автоматично LINQ-провайдером)
// ЗАТРАТИ ЧАСУ НА РОЗРОБКУ: Дуже малі. Підходить для гігантської бізнес-логіки.

public async Task<User?> GetUserByIdAsync(int id)
{
    // Немає ні запиту, ні підключення (все приховано)
    // Розробник мислить лише об'єктами
    return await _dbContext.Users.FirstOrDefaultAsync(u => u.Id == id);
}
```

### Коли і що використовувати?

`::field-group`
::field{name="EF Core" type="Вибір за замовчуванням"}
Використовуйте для 85% задач у сучасному вебі. Підтримка складних реляційних зв'язків типу (One-to-Many, Many-to-Many), міграції таблиць, відстеження змін (Change Tracking). "Турбота про інфраструктуру" знята з плечей розробника.
::
::field{name="Dapper" type="Micro-ORM"}
Використовуйте для CRUD сервісів, де критична мілісекундна продуктивність, або якщо ви маєте неймовірно складний SQL-запит (наприклад, 4 `LEFT JOIN`, віконні функції, `GROUP BY`), який EF Core генерує неефективно. Ви контролюєте кожен байт SQL, а Dapper просто бере на себе маппінг (перетворення таблиці в об'єкти C#).
::
::field{name="ADO.NET (Raw)" type="Low-Level"}
Сьогодні використовується для специфічних вузьких завдань.
Приклади:

1. Потокова видача звітів на гігабайти без з'їдання оперативної пам'яті (пам'ятаєте історію з початку?).
2. Використання специфічних функціональностей сервера типу `SqlBulkCopy` (вставка 100 тисяч рядків за 2 секунди).
3. Ви пишете власну бібліотеку, ORM, framework або абстракцію.

::
`::`

## 6. Практична Реалізація (Implementation Guide)

Давайте зламаємо "чорну скриньку" і напишемо наш найперший, повністю контрольований консольний застосунок для роботи з ADO.NET. Ми будемо використовувати `SQL Server` (або `LocalDB`, який часто встановлюється разом з Visual Studio).

::steps

### Крок 1: Створення проєкту та підключення пакета

Відкрийте термінал і створіть новий консольний проєкт на C#. Одразу ж додамо офіційний драйвер `Microsoft.Data.SqlClient`.
_Зверніть увагу: ми використовуємо `Microsoft.Data`, а не застарілий вбудований `System.Data.SqlClient`._

```bash
dotnet new console -n AdoNetFundamentals
cd AdoNetFundamentals
dotnet add package Microsoft.Data.SqlClient
```

### Крок 2: Розбираємо Connection String (Рядок підключення)

Як телефонному апарату потрібен номер телефону для дзвінка, так `SqlConnection` потрібен рядок підключення.
Збережіть його у змінній (у реальних застосунках він береться з файлу `appsettings.json` або змінних середовища "Environment Variables").

```csharp [Program.cs]
using Microsoft.Data.SqlClient; // [!code highlight]

// Формат: Server=Ім'яСервера;Database=Ім'яБД;User Id=Логін;Password=Пароль;
// Або для локальної Windows автентифікації (TrustServerCertificate для локалхосту у сучасних драйверах):
string connectionString = "Server=(localdb)\\mssqllocaldb;Database=master;Trusted_Connection=True;TrustServerCertificate=True";
```

### Крок 3: Виконання першої команди

Напишемо блок, який просто підключається до системної бази (`master`) і запитує поточну версію SQL Server через службову змінну `@@VERSION`.

```csharp [Program.cs]
using System;
using Microsoft.Data.SqlClient;

string connectionString = "Server=(localdb)\\mssqllocaldb;Database=master;Trusted_Connection=True;TrustServerCertificate=True";

Console.WriteLine("Намагаємось підключитися...");

try
{
    // 1. Створюємо підключення. Блок using гарантує безпечне закриття та виклик Dispose()
    await using var connection = new SqlConnection(connectionString);

    // 2. Фізично відкриваємо з'єднання (мережевий запит до БД)
    await connection.OpenAsync();
    Console.WriteLine("Успішно! Зв'язок встановлено.");

    // 3. Формуємо команду (Корабель, який повезе наш SQL-запит)
    await using var command = new SqlCommand("SELECT @@VERSION", connection);

    // 4. Виконуємо запит.
    // Так як @@VERSION повертає рівно одну колонку і один рядок - використовуємо ExecuteScalar!
    object result = await command.ExecuteScalarAsync();

    Console.WriteLine("\nВідомості про сервер:");
    Console.WriteLine(result.ToString());
}
catch (SqlException ex)
{
    // Specifcally catching Database-related errors
    Console.WriteLine($"Помилка бази даних: {ex.Message}");
}
catch (Exception ex)
{
    Console.WriteLine($"Загальна помилка: {ex.Message}");
}
```

::

### Анатомія коду (Декомпозиція)

- `using var connection` — це конструкція мови C# 8.0+. Під капотом, як тільки виконання дійде до кінця методу, C# самостійно вставить виклик `finally { connection.Dispose(); }`.
- `connection.OpenAsync()` — асинхронне відкриття гарантує, що головний потік застосунку не буде заблокований, якщо база даних знаходиться на іншому континенті або відповідає повільно.
- `command.ExecuteScalarAsync()` — ADO.NET надає кілька спеціалізованих методів виконання команд. `ExecuteScalar` ідеально підходить для математичних або агрегатних запитів типу `SELECT COUNT(*)` або повернення однієї єдиної комірки (Він бере 0-й рядок, 0-й стовпчик резалт-сету і просто ігнорує все інше, економлячи пам'ять).

## 7. Практика та Резюме (Practice & Summary)

Щоб знання стали навичками, вам потрібно "відчути" код власними руками.

### Завдання (Tasks)

::accordion

::accordion-item{label="🟢 Рівень 1 (Базовий: Ініціалізація та Зв'язок)" icon="i-lucide-dumbbell"}
**Завдання:**
Створіть застосунок, який тестує підключення.

1. Оголосіть рядок підключення з **помилковим** логіном або паролем.
2. Спробуйте викликати `Open()`.
3. Загорніть код у `try/catch` і зловіть `SqlException`. Виведіть в консоль власне номер помилки (`ex.Number`) і пояснення українською мовою для користувача типу "Шановний користувачу, сталась помилка автентифікації".

**Що вивчимо:** Робота з `SqlException`, який є джерелом всієї детальної інформації про відмови бази даних.
::

::accordion-item{label="🟡 Рівень 2 (Логіка: Builder Підключень)" icon="i-lucide-wrench"}
**Завдання:**
Рядок підключення конкатенувати (зшивати через `+`) вручну не рекомендується через ризик синтаксичної помилки (наприклад забути крапку з комою).

1. Знайдіть в офіційній документації клас `SqlConnectionStringBuilder`.
2. Створіть екземпляр цього класу.
3. Через властивості об'єкту (`builder.DataSource`, `builder.InitialCatalog` і т.д.) заповніть параметри підключення.
4. Виведіть згенерований `builder.ConnectionString` у консоль.

**Що вивчимо:** Утиліти, які надає ADO.NET для безпечної конфігурації.
::

::accordion-item{label="🔴 Рівень 3 (Архітектура: Міні-фабрика)" icon="i-lucide-building"}
**Завдання:**
Жоден нормальний проєкт не тримає `SqlConnection` прямо у `Program.cs`.

1. Створіть інтерфейс `IConnectionFactory`. В ньому має бути один метод: `Task<SqlConnection> CreateConnectionAsync()`.
2. Реалізуйте клас `SqlServerConnectionFactory : IConnectionFactory`.
3. Клас повинен приймати Connection String у своєму конструкторі (ніби він отримує його з Dependency Injection).
4. Метод `CreateConnectionAsync()` має не лише створювати `new SqlConnection()`, але й **одразу** викликати `OpenAsync()` та повертати підключення готовим до бою.

**Що вивчимо:** Архітектурний патерн `Factory`, правильне розділення відповідальності (Separation of Concerns). Ваші сервіси не повинні знати звідки береться підключення, вони лише споживають його.

### Резюме (Summary)

::card-group
::card{title="Підсумок" icon="i-heroicons-rectangle-group"}

1. **ADO.NET** — це найнижчий рівень абстракції, що знаходиться під усіма сучасними C# ORM системами.
2. Основні гравці: `Підключення` (Труба), `Команда` (Запит), `DataReader` (Кран з водою).
3. **Не забувайте про Dispose!** Ресурси бази даних (з'єднання) вмирають, коли програміст ігнорує виклик методів очищення або блоки `using`.
4. Вивчення ADO.NET - це ключ до усвідомлення того, інфраструктуру якого саме рівня і складності для вас "безкоштовно" будує Entity Framework Core.
