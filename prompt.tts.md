# Інструкція з оптимізації тексту для TTS (Audiobook)

Цей документ містить правила для підготовки україномовних технічних текстів до перетворення на аудіокнигу за допомогою систем синтезу мовлення (TTS).

## Головне правило: 100% збереження об'єму

**КРИТИЧНА ВИМОГА**: Заборонено скорочувати, підсумовувати або видаляти будь-які частини оригінального тексту. Об'єм має бути збережений повністю. Кожне речення, приклад, жарт чи технічне пояснення повинні залишитися. Оптимізація відбувається шляхом _зміни_ формулювань для кращого звучання, а не через видалення змісту.

## Правила оптимізації

### 1. Очищення від Markdown та Docus компонентів

**КРИТИЧНА ВИМОГА**: Видаляйте всі технічні конструкції Markdown та специфічні компоненти Docus, які не несуть смислового навантаження для слухача. Текст повинен бути чистим наративом.

-   **Docus-контейнери**: Видаляйте `::card`, `::alert`, `::code-group`, `::list`, `::link-card` тощо. Залишайте лише текст, що міститься всередині них.
-   **Посилання та медіа**:
    -   Видаляйте синтаксис посилань `[текст](url)`, залишаючи лише `текст`.
    -   Видаляйте зображення `![опис](шлях)`. Опис можна залишити лише як частину наративу, якщо це важливо (див. пункт "Обробка візуальних посилань").
-   **Код**: Програмний код ніколи не повинен зачитуватися як синтаксис (ніяких дужок, крапок з комою чи специфічних операторів). Код має бути перетворений у **наративний опис логіки** (див. розділ "Наративна трансформація коду").

### 2. Структурні маркери (Audio Navigation)

Додавайте явні маркери перед заголовками для кращої орієнтації слухача:

-   `# Заголовок 1` -> `Розділ: [Назва]`
-   `## Заголовок 2` -> `Підрозділ: [Назва]`
-   `### Заголовок 3` -> `Пункт: [Назва]`

### 3. Фонетична адаптація (Українізація термінів)

**КРИТИЧНА ВИМОГА**: Жодне англійське слово не повинно залишатися в оригінальному написанні в основному тексті. Все, що має бути озвучено, повинно бути написане українськими літерами так, як воно звучить (фонетично).

-   **Акроніми та скорочення**:
    -   `API` -> `Ей-Пі-Ай`
    -   `DDD` -> `Ді-Ді-Ді`
    -   `JSON` -> `Джі-Сон`
    -   `XML` -> `Екс-Ем-Ель`
    -   `HTTP` / `HTTPS` -> `Ейч-Ті-Ті-Пі` / `Ейч-Ті-Ті-Пі-Ес`
    -   `SQL` / `NoSQL` -> `Ес-Ку-Ель` / `Ноу-Ес-Ку-Ель`
    -   `UI/UX` -> `Ю-Ай slash Ю-Екс`
-   **Технічні терміни**:
    -   `Framework` -> `Фрейм-ворк`
    -   `Frontend` / `Backend` -> `Фронт-енд` / `Бек-енд`
    -   `Interface` -> `Інтер-фейс` (або просто по-українськи: інтерфейс)
    -   `Repository` -> `Репозиторій`
    -   `Service` -> `Сервіс`
    -   `Middleware` -> `Мідл-вер`
-   **Складні назви та моделі**: `Siemens KG86NAI3L` -> `Сіменс Кей-Джі-86-Ен-Ей-Ай-3-Ель`
-   **Англійська в дужках**: Завжди додавайте фонетичне звучання перед оригіналом у дужках:
    -   `(ubiquitous language)` -> `(юбіквітос ленгвідж — англійською ubiquitous language)`
    -   `Bounded Context` -> `Баунтед Контекст (англійською Bounded Context)`

### 4. Обробка візуальних посилань

Текст не повинен містити посилань на номери малюнків, які слухач не бачить. Перефразуйте їх у наративні переходи:

-   `як показано на рис. 4.9` -> `як ми бачимо на відповідному малюнку`
-   `див. мал. 1` -> `як ми бачимо на схемі`
-   Видаляйте прямі номери малюнків, якщо вони не несуть критичного змісту в аудіо, замінюючи на описові фрази.

### 5. Наративні списки

Перетворюйте марковані та нумеровані списки на природний розмовний потік:

-   Замість `- Пункт` використовуйте: `"По-перше,", "По-друге,", "Наступний момент,", "Перша причина,", "Ще одна важлива деталь,"`.
-   Це робить текст схожим на живу розповідь, а не на зачитування документа.

### 6. Пунктуація та темп

-   Використовуйте тире та крапки для створення пауз у логічних місцях.
-   Розбивайте занадто довгі та складні речення на коротші для легшого сприйняття на слух (але без втрати змісту!).

### 7. Збереження прикладів

Всі детальні приклади (історії про помідори, наукові теорії, купівлю холодильника тощо) мають бути збережені повністю. Вони критично важливі для розуміння контексту в аудіоформаті.

### 8. Технічні нюанси синтезу

-   **Наголоси**: Якщо програма некоректно визначає наголос, використовуйте символ `+` після наголошеного складу (наприклад, `вип+ишіть`).
-   **Контекст речень**: Уникайте окремих слів. Текст, що складається з одного слова, може синтезуватися з артефактами. Завжди пишіть повноцінні речення.
-   **Вербалізація**: Всі цифри, символи та акроніми повинні бути переведені в словесну форму українською мовою. Наприклад:
    -   `1893` -> `тисяча вісімсот дев'яносто третього`
    -   `&` -> `енд` (або `і`)
    -   `v14.3.0` -> `версії чотирнадцять три нуль`
    -   `API` -> `Ей-Пі-Ай`

### 9. Наративна трансформація коду

Цей розділ описує як перетворювати кодові блоки на текст, який приємно слухати.

-   **Заборона синтаксису**: Не вимовляйте `{}`, `()`, `;`, `=>`, `[]`.
-   **Опис наміру**: Замість "функція ікс приймає ігрек" кажіть "ми створюємо дію під назвою ікс, яка працює з даними ігрек".
-   **Логічні переходи**: Використовуйте фрази: "Уявімо...", "Тут ми бачимо...", "Наступним кроком ми...", "Це дозволяє нам...".
-   **Порівняння**: Якщо текст показує еволюцію коду, описуйте це як процес: "Раніше ми робили так... але тепер, щоб покращити систему, ми змінюємо підхід на...".
-   **Анонімізація складних назв**: Якщо в коді є складні змінні, спрощуйте їх у мовленні (наприклад, `userRepository` -> `репозиторій користувачів`).

**Приклад трансформації коду**:

-   **Оригінал**:
    ```csharp
    public void Process(Order order) {
        if (order.IsPaid) {
            shipmentService.Ship(order);
        }
    }
    ```
-   **Оптимізовано для TTS**:
    > Розглянемо приклад обробки замовлення. Ми створюємо процес, який перевіряє: чи було замовлення оплачене? Якщо так, ми звертаємося до сервісу доставки та ініціюємо відправку цього замовлення клієнту.

## Приклад трансформації

**Оригінал**:

**Оптимізовано для TTS**:
